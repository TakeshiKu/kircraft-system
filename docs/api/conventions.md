# API Conventions

## 1. Назначение документа

Документ задает единые правила проектирования и оформления API системы Kircraft.

Цели документа:
- обеспечить единообразие API
- упростить разработку и сопровождение
- исключить неоднозначности при интеграции

Все модули API обязаны следовать данным соглашениям.

---

## 2. Базовые соглашения

### 2.1 Base URL

Все методы API публикуются с базовым префиксом:

/api/v1

### 2.2 Протокол

- HTTP/HTTPS
- REST-подход

### 2.3 Формат обмена

- JSON
- Content-Type: application/json

---

## 3. Правила именования

### 3.1 Endpoint

- используются существительные
- для коллекций используется множественное число
- используется нижний регистр
- слова в пути разделяются дефисом

Примеры:
- /cart
- /cart/items
- /orders
- /orders/{order_id}

Для action endpoints допускается использование вложенного пути с глаголом действия, если операция не выражается стандартной CRUD-семантикой.

Примеры:
- /orders/{order_id}/cancel
- /payments/{payment_id}/capture

Правила:
- action endpoint используется только для явного бизнес-действия
- имя действия должно быть коротким, однозначным и в нижнем регистре
- action endpoint не должен подменять обычные операции создания, чтения, обновления или удаления

### 3.2 Path-параметры

- используются в формате snake_case
- отражают идентификатор сущности или параметр маршрута

Примеры:
- order_id
- item_id

### 3.3 Query-параметры

- используются в формате snake_case
- применяются для фильтрации, сортировки и пагинации

Примеры:
- /orders?status=paid
- /products?limit=20&offset=0

### 3.4 JSON-поля

- используются в формате snake_case
- имена полей должны быть однозначными и стабильными
- опубликованные поля не переименовываются без изменения версии API

Пример:
{
  "order_id": "123",
  "created_at": "2026-01-01T12:00:00Z"
}

---

## 4. Формат данных

### 4.1 Дата и время

- используется формат ISO 8601
- время передается в UTC

Пример:
2026-01-01T12:00:00Z

### 4.2 Денежные значения

- денежные значения передаются в минимальных единицах
- тип значения: integer

Пример:
{
  "amount": 150000,
  "currency": "RUB"
}

### 4.3 Булевы значения

- используются значения true / false
- строковые представления булевых значений не допускаются

### 4.4 Enum-поля

- используются строковые значения
- набор допустимых значений фиксируется в контракте метода
- добавление новых значений допускается без смены версии API при сохранении обратной совместимости

---

## 5. Request / Response

### 5.1 Общие правила

- request должен содержать только необходимые поля
- response должен быть минимальным, но достаточным
- поля не должны дублировать друг друга без необходимости

### 5.2 Формат успешного ответа

Ответ должен быть консистентным в рамках одного модуля и типа операций.

Рекомендуемая структура:
{
  "data": {},
  "meta": {}
}

Правила:
- data содержит основной результат операции
- meta содержит служебную информацию при необходимости
- meta может отсутствовать, если служебные данные не требуются
- в успешном ответе поле error отсутствует

Дополнительно:
- meta используется для служебной информации, не относящейся к бизнес-сущности
- meta не должен дублировать данные из data
- структура meta должна быть предсказуемой и согласованной между методами

Допустимые типы данных в meta:
- request_id — идентификатор запроса
- параметры пагинации (limit, offset, total)
- служебные флаги, влияющие на обработку ответа

Недопустимо:
- размещение бизнес-данных
- дублирование структуры data
- передача внутренней технической информации backend

### 5.3 Пустой ответ

При успешной операции без тела ответа используется:
HTTP 204 No Content

### 5.4 Ответ при создании ресурса (201 Created)

При успешном создании ресурса API возвращает:

- HTTP статус: 201 Created
- тело ответа с созданным ресурсом

Формат ответа:
{
  "data": {
    // созданный ресурс
  },
  "meta": {}
}

Правила:
- response должен содержать актуальное состояние созданного ресурса
- минимально должен присутствовать идентификатор ресурса
- допускается возвращать полный объект, если это не приводит к избыточности
- поле error в успешном ответе отсутствует

### 5.5 Общий контракт ответа

API использует два типа top-level ответа:

#### Успешный ответ
{
  "data": {},
  "meta": {}
}

#### Ошибка
{
  "error": {
    "code": "string_code",
    "message": "string",
    "details": {},
    "request_id": "uuid"
  }
}

Правила:
- структура ответа зависит от результата выполнения (success / error)
- data и error не используются одновременно
- клиент обязан обрабатывать оба варианта ответа

---

## 6. Пагинация

Для коллекций используется offset-based пагинация.

Параметры:
- limit
- offset

Пример ответа:
{
  "data": [],
  "meta": {
    "limit": 20,
    "offset": 0,
    "total": 100
  }
}

---

## 7. Фильтрация и сортировка

### 7.1 Фильтрация

Пример:
- /orders?status=paid

### 7.2 Сортировка

Пример:
- /orders?sort=created_at,-price

Правила:
- поле без префикса означает сортировку по возрастанию
- префикс "-" означает сортировку по убыванию
- множественная сортировка допускается

---

## 8. HTTP статус-коды

- 200 — успешный запрос
- 201 — ресурс создан
- 204 — успешный запрос без тела
- 400 — ошибка запроса
- 401 — не авторизован
- 403 — доступ запрещен
- 404 — ресурс не найден
- 409 — конфликт состояния
- 422 — ошибка бизнес-валидации
- 500 — внутренняя ошибка сервера

___

## 9. HTTP методы

В API используются следующие HTTP-методы:

### GET
Используется для получения данных.
Правила:
- не изменяет состояние системы
- не должен иметь побочных эффектов на бизнес-уровне
- используется для получения ресурса, списка ресурсов, результатов расчета, если не создается новый серверный объект

### POST
Используется для:
- создания нового ресурса
- выполнения операции, которая приводит к изменению состояния
- вызова действий, не укладывающихся в семантику PATCH или DELETE

Примеры:
- создание заказа
- создание платежа
- выполнение серверной операции, требующей отдельного action endpoint

### PATCH
Используется для частичного обновления существующего ресурса.

Правила:
- request содержит только изменяемые поля
- отсутствие поля не означает его удаление
- поведение null фиксируется в контракте метода

### DELETE
Используется для удаления ресурса или удаления связи ресурса с другой сущностью.

Правила:
- должен применяться только там, где удаление действительно является частью публичного контракта
- если физическое удаление не допускается, в контракте явно фиксируется логика отмены, деактивации или снятия связи

### PUT
По умолчанию не используется.
Если полный replacement ресурса потребуется в будущем, его использование должно быть явно зафиксировано в контракте конкретного метода.

## 10. Идемпотентность

Идемпотентность обязательна для критичных операций:
- создание платежа
- создание заказа

Используется заголовок:
- Idempotency-Key

Требования:
- повторный запрос с тем же ключом должен возвращать согласованный результат
- ключ обрабатывается на стороне backend
- область действия ключа определяется комбинацией:
  - HTTP-метод
  - endpoint
  - клиентский контекст
- повторное использование одного и того же ключа для другого payload в рамках той же области действия должно приводить к ошибке
- срок хранения и правила истечения ключа определяются внутренней реализацией backend и не являются частью публичного API-контракта

---

## 11. Трассировка запросов

Используется заголовок:
- X-Request-Id

Требования:
- если передан клиентом — используется
- если отсутствует — генерируется backend
- возвращается в заголовке ответа

Дополнительно:
- request_id должен присутствовать в теле ответа:
  - в успешных ответах — в meta
  - в ошибках — в объекте error

Пример успешного ответа:
{
  "data": {},
  "meta": {
    "request_id": "uuid"
  }
}

Пример ошибки:
{
  "error": {
    "code": "string_code",
    "message": "string",
    "details": {},
    "request_id": "uuid"
  }
}

---

## 12. Частичные обновления

Используется метод PATCH.

Правила:
- request содержит только изменяемые поля
- отсутствие поля не означает его удаление
- поведение null фиксируется в контракте метода

---

## 13. Nullable и обязательные поля

- обязательные поля явно указываются в контракте
- отсутствие поля и null — разные состояния
- null используется только при явной необходимости

Правила интерпретации:
- отсутствие поля в request означает, что значение не передано
- null в request означает явную передачу пустого значения, если это допускается контрактом метода
- отсутствие поля в response означает, что поле не входит в контракт данного ответа
- null в response означает, что поле входит в контракт, но значение отсутствует

Использование null должно быть минимальным и явно зафиксированным в контракте конкретного метода.

---

## 14. Эволюция API

- новые поля могут добавляться без смены версии API
- изменение или удаление поля требует новой версии
- поля не переименовываются в рамках версии

---

## 15. Заголовки

Используются:
- Content-Type
- Idempotency-Key
- X-Request-Id

Дополнительные заголовки фиксируются в контрактах методов.

---

## 16. Безопасность (общие принципы)

- backend не должен доверять данным клиента
- доступ к ресурсам ограничивается правами пользователя
- клиент может получать доступ только к своим данным
- критичные операции должны быть защищены (идемпотентность, проверки состояния)
- чувствительные данные не должны логироваться или передаваться без необходимости

Детальные требования безопасности описываются в отдельном документе:
docs/api/security.md

---

## 17. Ограничения и допущения

- bulk-операции не поддерживаются (если не указано иное)
- клиент может повторять запрос при временных сбоях
- backend должен быть устойчив к повторным запросам
- внешние интеграции могут быть недоступны и должны учитываться в логике

Дополнительно:
- повтор запроса допустим при временных ошибках связи и ответах 5xx
- повтор non-idempotent операций без Idempotency-Key не гарантирует безопасный результат
- правила повторной отправки webhook и интеграционных вызовов фиксируются в контрактах соответствующих методов или интеграционных документов
