# API Security

## 1. Назначение документа

Документ описывает правила безопасности backend API системы Kircraft.

Цели:
- определить модель аутентификации и авторизации
- зафиксировать правила доступа к данным
- обеспечить защиту критичных операций
- установить единые требования к интеграциям и webhook
- определить базовые ограничения и принципы защиты API

Документ задает архитектурные принципы и не описывает детали внутренней реализации.

---

## 2. Общая модель безопасности

API строится на следующих принципах:

- backend не доверяет данным клиента
- каждый запрос выполняется в контексте аутентифицированного субъекта или явно определенного анонимного сценария
- доступ к ресурсам определяется на основе прав и владения
- чувствительные операции требуют дополнительных проверок
- интеграции с внешними системами проходят в изолированном контексте
- безопасность обеспечивается на всех уровнях взаимодействия: транспорт, аутентификация, авторизация, валидация, аудит

---

## 3. Границы доверия

В системе выделяются следующие внешние контуры взаимодействия:

- клиентские интерфейсы (Web, Telegram-бот, Mini App)
- административный интерфейс
- внешние сервисы и интеграции
- webhook от внешних провайдеров

Для каждого контура действуют отдельные правила аутентификации, авторизации и валидации.

Данные, поступающие из любого внешнего контура, считаются недоверенными до прохождения проверок на стороне backend.

---

## 4. Транспортный уровень

Все взаимодействие с API должно выполняться только по HTTPS.

Правила:
- передача данных по незашифрованному HTTP не допускается
- чувствительные данные не должны передаваться вне защищенного канала
- требования к TLS и сетевой инфраструктуре определяются на уровне инфраструктуры и не являются частью публичного API-контракта

---

## 5. Аутентификация

Аутентификация определяет, кто выполняет запрос.

### 5.1 Типы субъектов

В системе выделяются следующие типы субъектов:

- клиент (пользователь системы)
- административный пользователь
- внешний сервис (интеграции, webhook)

Каждый тип субъекта проходит свою процедуру аутентификации.

---

### 5.2 Модель идентификации пользователя

В системе используется единая сущность User.

Пользователь может иметь несколько внешних идентификаторов:
- Telegram
- Web
- другие каналы

Правила:
- внутренний идентификатор пользователя является основным
- внешние идентификаторы используются только для входа и связывания канала с пользователем
- бизнес-данные (заказы, корзина, профиль) всегда привязаны к User
- архитектура не должна зависеть только от одного внешнего идентификатора, например telegram_user_id

---

### 5.3 Требования к механизму аутентификации

Механизм аутентификации должен:
- быть единым и согласованным на уровне системы
- поддерживать клиентские интерфейсы и административный доступ
- позволять изолировать пользовательский, административный и сервисный контекст
- допускать последующее развитие без перелома публичного API

---

### Точка выбора

Требуется определить базовый механизм аутентификации:

Варианты:
1. Token-based модель
2. Session-based модель

Рекомендация:
- использовать token-based модель как базовую
- предусмотреть возможность расширения для Web-клиента при необходимости

Дополнительные замечания:
- если используется session-based модель для Web, должны быть отдельно описаны правила защиты от CSRF
- если используется token-based модель, должны быть отдельно зафиксированы правила хранения, обновления и отзыва токенов

---

## 6. Авторизация

Авторизация определяет, что субъект может делать.

### 6.1 Общие правила

- доступ к ресурсам проверяется только на backend
- клиент не может получать доступ к чужим данным
- любые идентификаторы в запросе валидируются через контекст субъекта
- факт знания идентификатора ресурса не дает права на доступ к нему

---

### 6.2 Владение ресурсом

Правила:

- пользователь имеет доступ только к своим ресурсам:
  - корзина
  - заказы
  - профиль

- доступ определяется не по переданному ID, а по связи ресурса с User

Пример:
- запрос `/orders/{order_id}`
- backend обязан проверить, принадлежит ли заказ текущему пользователю

---

### 6.3 Административный доступ

- административные методы изолированы в отдельном контуре API
- административный доступ требует отдельной аутентификации
- обычный клиентский контекст не должен давать доступ к административным операциям
- административные действия должны выполняться по принципу минимально необходимых прав

---

### Точка выбора

Модель прав доступа:

Варианты:
1. Простая модель `user / admin`
2. Расширяемая ролевая модель

Рекомендация:
- начать с простой модели
- заложить возможность расширения до ролевой модели без изменения базовой архитектуры

---

## 7. Контекст запроса

Каждый запрос обрабатывается в контексте:

- идентификатора субъекта
- типа субъекта (`client / admin / service`)
- request_id для трассировки
- набора прав, доступных в рамках текущего запроса

Контекст формируется на backend и не может быть задан клиентом напрямую.

---

## 8. Принцип минимальных привилегий

Во всех сценариях должен соблюдаться принцип минимально необходимых прав.

Правила:
- субъект получает только те права, которые необходимы для выполнения конкретного действия
- сервисные учетные данные не должны использоваться в пользовательском контексте
- административные полномочия не должны применяться для обычных клиентских операций
- расширение прав должно быть явным и контролируемым

---

## 9. Защита критичных операций

К критичным операциям относятся:

- создание заказа
- создание платежа
- изменение статуса заказа
- административные действия
- обработка webhook
- любые действия, приводящие к финансовым, статусным или необратимым изменениям

---

### 9.1 Общие требования

- операции должны быть идемпотентными, если сценарий допускает повторную отправку
- состояние должно проверяться перед выполнением операции
- повторные запросы не должны приводить к неконсистентному состоянию
- backend не должен полагаться только на состояние, переданное клиентом

---

### 9.2 Защита от повторной отправки

- используется `Idempotency-Key` для операций, где это требуется контрактом API
- повторная отправка не должна создавать дублирующие операции
- изменение payload при том же ключе в рамках одной области действия должно приводить к ошибке
- правила идемпотентности определяются в `conventions.md` и соблюдаются всеми критичными методами

---

## 10. Webhook и внешние вызовы

Webhook является отдельным типом взаимодействия и не связан с пользовательским контекстом.

### 10.1 Общие правила

- webhook рассматривается как запрос от внешнего сервиса
- входящие данные webhook считаются недоверенными
- каждый webhook должен проходить аутентификацию и валидацию
- обработка webhook должна быть изолирована от клиентского и административного контекста

---

### 10.2 Проверка подлинности webhook

Допустимые механизмы:
- подпись запроса
- секретный ключ
- дополнительные сетевые ограничения

Рекомендации:
- подпись запроса должна использоваться как основной механизм проверки подлинности
- IP whitelist может использоваться как дополнительный слой защиты, но не как единственный механизм

---

### 10.3 Идемпотентность webhook

- webhook может приходить повторно
- webhook может быть доставлен позже ожидаемого момента
- обработка webhook должна быть идемпотентной
- повторные события не должны ломать состояние системы

---

## 11. Работа с внешними интеграциями

Интеграции с платежными, логистическими и иными сервисами:

- выполняются от имени backend
- изолированы от пользовательского контекста
- используют отдельные учетные данные и правила доступа
- не должны раскрывать клиенту внутренние детали взаимодействия с провайдером

Ошибки интеграций должны возвращаться в безопасном и стандартизированном виде в соответствии с `errors.md`.

---

## 12. Валидация и обработка входящих данных

### 12.1 Общие правила

- все входящие данные валидируются на backend
- проверяются типы, диапазоны, формат, обязательность и допустимость значений
- недопустимые данные отклоняются до выполнения бизнес-операции

---

### 12.2 Защита от недоверенных данных

- backend не должен использовать клиентские данные без проверки
- входящие идентификаторы, статусы, суммы и иные чувствительные поля должны проверяться на допустимость и контекст
- внутренние переходы состояний не должны определяться только значением, пришедшим от клиента

---

## 13. Защита данных

### 13.1 Общие правила

- не передавать чувствительные данные без необходимости
- не логировать чувствительные данные без необходимости
- доступ к данным должен быть ограничен в соответствии с ролью и владением ресурсом

---

### 13.2 Чувствительные данные

К чувствительным данным относятся, в том числе:
- токены и иные секреты доступа
- платежные данные
- персональные данные
- внутренние служебные учетные данные интеграций

Такие данные:
- не должны раскрываться в ошибках API
- не должны возвращаться клиенту без прямой необходимости
- не должны попадать в логи в открытом виде

---

## 14. Политика междоменных запросов

Для Web-клиентов политика междоменных запросов должна быть явно определена.

Правила:
- список допустимых origin должен контролироваться backend и инфраструктурой
- небезопасная политика междоменных запросов не допускается
- настройки доступа для браузерных клиентов должны соответствовать выбранной модели аутентификации

Если для Web используется cookie-based аутентификация, требования к CORS и CSRF должны быть описаны дополнительно.

---

## 15. Ограничение доступа и злоупотреблений

### 15.1 Rate limiting

Backend должен иметь механизмы ограничения частоты запросов для:
- защиты от злоупотреблений
- ограничения автоматизированных атак
- защиты чувствительных и дорогих по стоимости операций

Детали реализации не являются частью публичного API-контракта.

---

### 15.2 Защита от brute-force

Должны быть предусмотрены меры против перебора:
- ограничение частоты попыток аутентификации
- временные блокировки или иные защитные меры при подозрительной активности

---

## 16. Логирование и аудит

Система должна обеспечивать трассировку и аудит значимых событий безопасности.

Должны фиксироваться:
- попытки аутентификации
- ошибки авторизации
- административные действия
- обработка критичных webhook и интеграционных событий
- операции, изменяющие статус заказа или оплаты

Правила:
- журналы не должны содержать чувствительные данные в открытом виде
- состав и глубина аудита определяются внутренней реализацией
- требования к `request_id` и трассировке описаны в `conventions.md`

---

## 17. Ошибки безопасности

- ошибки безопасности не должны раскрывать внутреннюю реализацию
- сообщения должны быть безопасными для отображения клиенту
- внутренние детали, причины отказа и диагностическая информация должны оставаться на стороне backend
- формат ошибок должен соответствовать `errors.md`

---

## 18. Эволюция безопасности

- требования безопасности могут усиливаться без изменения версии API, если это не ломает публичный контракт
- новые защитные механизмы могут добавляться постепенно
- развитие безопасности не должно приводить к смешению пользовательского, административного и сервисного контекста
- новые интеграции и каналы доступа должны включаться в существующую модель безопасности, а не обходить ее
