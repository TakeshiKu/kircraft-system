# Order API

## 1. Назначение модуля

Order API предназначен для работы с заказами клиента в системе Kircraft.

Модуль обеспечивает:
- создание заказа на основании активной корзины клиента
- получение списка заказов клиента
- получение информации по конкретному заказу
- отмену заказа клиентом в допустимых статусах

Order API используется клиентскими интерфейсами системы:
- Web
- Telegram-бот
- Mini App
- другие клиентские каналы

Все операции выполняются от имени текущего аутентифицированного клиента.

---

## 2. Границы ответственности модуля

Order API отвечает за:
- создание заказа из активной корзины клиента
- фиксацию snapshot-данных заказа на момент оформления
- предоставление клиенту информации о его заказах
- отмену заказа клиентом в допустимых статусах

Order API не отвечает за:
- управление составом корзины
- изменение состава заказа после создания
- расчет и выбор доставки как отдельный процесс
- инициацию и сопровождение оплаты
- административное управление заказами
- перевод заказа по внутренним производственным статусам

---

## 3. Связанные сущности

Модуль использует следующие сущности модели данных:
- `orders`
- `order_items`
- `order_item_selected_values`

Модуль зависит от следующих сущностей:
- `customers`
- `carts`
- `cart_items`
- `cart_item_selected_values`

Связь с корзиной:
- заказ создается только из активной корзины клиента
- используется вся корзина целиком
- частичное оформление корзины не поддерживается
- после успешного создания заказа корзина получает статус `converted`
- из одной корзины создается только один заказ

Поведение корзины после создания заказа:
- корзина в статусе `converted` не может использоваться для повторного оформления заказа
- новая активная корзина создается при следующем добавлении товара клиентом

Связь с оплатой:
- после успешного создания заказа клиент переходит к оплате
- логика оплаты не входит в область ответственности Order API
- взаимодействие с оплатой описывается в `payment-api.md`

## 4. Правила работы заказа

### 4.1 Общие правила

- Order API работает только в контексте текущего аутентифицированного клиента
- все операции выполняются в рамках `customer_id` текущего пользователя
- клиент не может получать доступ к чужим заказам
- клиент не может изменять состав заказа после его создания
- создание заказа доступно только для активной корзины текущего клиента

### 4.2 Правила создания заказа

- заказ создается только из активной корзины клиента
- заказ создается по всей корзине целиком
- частичное оформление отдельных позиций корзины не поддерживается
- если заказ уже был создан из текущей корзины, повторное создание заказа не допускается
- при повторном вызове создания заказа возвращается ошибка `cart_already_converted`
- после успешного создания заказа корзина получает статус `converted`

### 4.3 Правила валидации перед созданием заказа

Перед созданием заказа backend проверяет:
- активная корзина существует
- корзина не пуста
- все позиции корзины доступны для оформления
- выбранные параметры товаров валидны
- обязательные контактные данные заполнены (`customer_name`, `phone`), `email` является необязательным
- доставка выбрана и валидна
- итоговая стоимость заказа актуальна

Если хотя бы одно условие не выполнено:
- заказ не создается
- клиент получает ошибку
- корзина сохраняется в текущем состоянии

### 4.4 Правила фиксации данных заказа

При успешном создании заказа backend фиксирует snapshot-данные:
- имя клиента
- телефон клиента
- email клиента (если был указан)
- состав заказа
- цену каждой позиции
- выбранные параметры товаров
- стоимость доставки
- итоговую стоимость заказа

После создания заказа дальнейшие изменения исходной корзины или каталога не должны изменять уже созданный заказ.

### 4.5 Стартовый статус заказа

Заказ, созданный в клиентском checkout flow, получает статус `awaiting_payment`.

Статус `created` не используется как стартовый статус в клиентском сценарии оформления заказа.

Заказ в статусе `awaiting_payment` может быть автоматически отменен системой по таймауту. Конкретное время таймаута в рамках данного документа не фиксируется.

Статусы `created` и `draft` являются внутренними техническими статусами и не используются как рабочие статусы в клиентском checkout flow.

### 4.6 Правила изменения заказа

- клиентский API не поддерживает изменение состава заказа
- клиентский API не поддерживает изменение количества позиций заказа
- клиентский API не поддерживает изменение выбранных параметров товаров в заказе
- если клиенту нужно изменить состав покупки, он должен отменить заказ, если это допустимо, и оформить новый

### 4.7 Правила отмены заказа

Клиент может отменить заказ только в статусах:
- `awaiting_payment`
- `needs_clarification`, если такой статус используется в процессе обработки заказа

Клиент не может отменить заказ в статусах:
- `paid`
- `in_progress`
- `shipped`
- `cancelled`
- `rejected`

При успешной отмене заказа:
- заказ получает статус `cancelled`

### 4.8 Статусы заказа в клиентском API

В рамках клиентского API используются статусы:
- `draft`
- `awaiting_payment`
- `needs_clarification`
- `paid`
- `in_progress`
- `shipped`
- `cancelled`
- `rejected`

Статусы `created` и `draft` могут существовать в общей модели, но не используются как рабочие стартовые статусы в клиентском checkout flow.

## 5. Endpoint overview

Модуль включает следующие методы:
- `POST /order` - создать черновик заказа (`draft`) без корзины (MVP)
- `PATCH /order/delivery` - сохранить выбранный вариант доставки в заказ в статусе `draft`
- `POST /orders` - создать заказ из активной корзины клиента
- `GET /orders` - получить список заказов текущего клиента
- `GET /orders/{order_id}` - получить информацию по конкретному заказу
- `POST /orders/{order_id}/cancel` - отменить заказ текущего клиента в допустимом статусе

---

## 6. Общие правила

Допустимые статусы заказа:
- `draft`
- `awaiting_payment`
- `needs_clarification`
- `paid`
- `in_progress`
- `shipped`
- `cancelled`
- `rejected`

Поле `status` имеет тип `string` и принимает только значения из списка допустимых статусов заказа.

Допустимые значения `source_channel`:
- `telegram`
- `web`
- `mini_app`

Общие ограничения:
- из одной корзины может быть создан только один заказ
- backend должен гарантировать, что повторное создание заказа из одной корзины невозможно
- `order_id` является уникальным идентификатором заказа в системе
- все денежные значения в рамках Order API передаются в валюте `RUB`
- если в ответах присутствуют поля `currency` / `delivery_currency`, они должны быть консистентны с валютой `RUB` (все такие поля принимают значение `RUB`)
- поле `email` в заказе является необязательным: отсутствие `email` не блокирует создание заказа

## 7. Контракты методов

### 7.1 Создание заказа

#### Метод и путь

`POST /orders`

#### Назначение

Создает заказ на основании активной корзины текущего клиента.

Метод использует:
- активную корзину клиента
- ранее введенные контактные данные
- выбранные параметры доставки
- комментарий клиента (если был указан при оформлении)

После успешного создания заказа:
- создается заказ
- фиксируются данные заказа на момент оформления
- корзина получает статус `converted`
- заказ получает статус `awaiting_payment`

#### Request

Body отсутствует.

#### Правила request

- запрос выполняется только в контексте текущего аутентифицированного клиента
- используется только активная корзина текущего клиента
- повторный вызов метода для уже конвертированной корзины не допускается
- операция создания заказа является идемпотентной, используйте заголовок `Idempotency-Key` (повторный запрос с тем же ключом возвращает согласованный результат)

#### Response 201

```json
{
  "data": {
    "order_id": "order_001",
    "status": "awaiting_payment",
    "source_channel": "telegram",
    "customer_name": "Владимир",
    "phone": "+79990000000",
    "email": "vladimir@example.com",
    "comment": "Позвонить перед отправкой",
    "items": [
      {
        "order_item_id": "order_item_001",
        "product_id": "product_001",
        "product_name": "Обложка для паспорта",
        "price": 150000,
        "quantity": 2,
        "line_total": 300000,
        "selected_values": [
          {
            "parameter_id": "parameter_001",
            "parameter_name": "Цвет",
            "parameter_value_id": "value_005",
            "value_name": "Черный"
          }
        ]
      }
    ],
    "delivery": {
      "city": "Москва",
      "delivery_type": "pickup_point",
      "delivery_provider": "cdek",
      "pickup_point_id": "PVZ_001",
      "pickup_point_name": "СДЭК ПВЗ",
      "pickup_point_address": "Москва, ул. Пример, д. 1",
      "delivery_price": 50000,
      "delivery_currency": "RUB"
    },
    "totals": {
      "items_total": 300000,
      "delivery_price": 50000,
      "total_price": 350000,
      "currency": "RUB"
    },
    "created_at": "2026-03-19T18:30:00Z",
    "updated_at": "2026-03-19T18:30:00Z"
  },
  "meta": {
    "request_id": "uuid"
  }
}
```

#### Правила ответа

- ответ содержит созданный заказ в актуальном состоянии
- возвращаются зафиксированные данные заказа на момент оформления
- поле `email` возвращается как строка, если email был указан при checkout, иначе `null`
- цены передаются в минимальных денежных единицах
- `selected_values` всегда передается массивом и может быть пустым
- поле `paid_at` отсутствует до момента оплаты заказа
- после успешного ответа исходная корзина считается конвертированной и не может использоваться повторно для создания заказа

#### Возможные ошибки

- `401 unauthorized` - пользователь не аутентифицирован
- `409 cart_not_active` - отсутствует активная корзина
- `409 cart_already_converted` - заказ уже был создан из данной корзины
- `422 validation_error` - некорректные данные (контакты, доставка, параметры)
- `422 item_not_available` - в корзине есть недоступные товары
- `500 internal_error` - внутренняя ошибка сервера

### 7.2 Сохранение доставки в черновик заказа

#### Метод и путь

`PATCH /order/delivery` (полный путь: `PATCH /api/v1/order/delivery`)

#### Назначение

Сохраняет в заказ выбранный вариант доставки (ПВЗ, цена, сроки). Доступно только для заказа в статусе `draft`. Повторный вызов перезаписывает данные доставки (MVP). Не вызывает провайдера доставки и не пересчитывает тариф на сервере.

#### Request

```json
{
  "order_id": "uuid",
  "delivery_option": {
    "pickup_point_id": "string",
    "pickup_point_name": "string",
    "pickup_point_address": "string",
    "delivery_eta_min_days": 0,
    "delivery_eta_max_days": 0,
    "delivery_price": 0,
    "delivery_currency": "RUB"
  }
}
```

#### Правила request

- все поля обязательны
- `delivery_price`, `delivery_eta_min_days`, `delivery_eta_max_days` — целые числа ≥ 0
- остальные поля — непустые строки (после trim)

#### Response 200

```json
{
  "data": {
    "order_id": "uuid"
  },
  "meta": {
    "request_id": "uuid"
  }
}
```

#### Возможные ошибки

- `401 unauthorized` — не аутентифицирован
- `400 validation_error` — невалидное тело запроса
- `404 order_not_found` — заказ не найден или не принадлежит клиенту
- `409 order_invalid_state` — заказ не в статусе `draft`
- `500 order_update_failed` — ошибка сохранения в БД

### 7.3 Получение списка заказов клиента

#### Метод и путь

`GET /orders`

#### Назначение

Возвращает список заказов текущего клиента.

#### Query-параметры

- `limit` - необязательный, количество записей на страницу
- `offset` - необязательный, смещение
- `status` - необязательный, фильтр по статусу заказа

#### Правила request

- список возвращается только для текущего аутентифицированного клиента
- фильтрация по `customer_id` не поддерживается
- параметр `status` должен соответствовать допустимому статусу заказа

#### Response 200

```json
{
  "data": [
    {
      "order_id": "order_001",
      "status": "awaiting_payment",
      "items_total": 300000,
      "delivery_price": 50000,
      "total_price": 350000,
      "created_at": "2026-03-19T18:30:00Z",
      "updated_at": "2026-03-19T18:30:00Z"
    },
    {
      "order_id": "order_002",
      "status": "paid",
      "items_total": 150000,
      "delivery_price": 50000,
      "total_price": 200000,
      "created_at": "2026-03-18T10:15:00Z",
      "updated_at": "2026-03-18T11:00:00Z"
    }
  ],
  "meta": {
    "request_id": "uuid",
    "limit": 20,
    "offset": 0,
    "total": 2
  }
}
```

#### Правила ответа

- возвращаются только заказы текущего клиента
- список может быть пустым
- по умолчанию заказы возвращаются в порядке убывания `created_at`

#### Возможные ошибки

- `401 unauthorized` - пользователь не аутентифицирован
- `400 invalid_request` - некорректные параметры запроса
- `500 internal_error` - внутренняя ошибка сервера

### 7.5 Отмена заказа клиентом

#### Метод и путь

`POST /orders/{order_id}/cancel`

#### Назначение

Отменяет заказ текущего клиента, если его статус допускает отмену.

#### Request

Body отсутствует.

#### Правила request

- `order_id` должен относиться к заказу текущего клиента
- клиент может отменить заказ только в статусах:
  - `draft`
  - `awaiting_payment`
  - `needs_clarification`
- при попытке отмены в недопустимом статусе (включая уже `cancelled`) возвращается ошибка `order_not_cancellable`

#### Response 200

```json
{
  "data": {
    "order_id": "order_001",
    "status": "cancelled"
  },
  "meta": {
    "request_id": "uuid"
  }
}
```

#### Правила ответа

- ответ подтверждает отмену (`order_id` + `status: cancelled`); полный снимок заказа — через `GET /api/v1/orders/{order_id}`
- повторная отмена заказа в статусе `cancelled` возвращает ошибку `order_not_cancellable`

#### Возможные ошибки

- `400 invalid_request` — пустой или некорректный `order_id` в пути
- `401 unauthorized` - пользователь не аутентифицирован
- `404 order_not_found` - заказ не найден или не принадлежит пользователю
- `409 order_not_cancellable` - заказ находится в статусе, не допускающем отмену
- `500 internal_error` - внутренняя ошибка сервера
