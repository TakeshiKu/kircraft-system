# Payment API

## 1. Назначение модуля

Payment API предназначен для работы с оплатой заказов клиента в системе Kircraft.

Модуль обеспечивает:
- создание попытки оплаты для заказа клиента
- получение информации о платеже
- обработку результата оплаты
- синхронизацию статуса платежа с внешней платежной системой
- обновление статуса заказа по результату оплаты

Payment API используется:
- клиентскими интерфейсами системы
- Backend API Kircraft
- внешней платежной системой YooKassa

Клиентские интерфейсы системы могут включать:
- Web
- Telegram-бот
- Mini App
- другие клиентские каналы

Непосредственное взаимодействие с API YooKassa выполняется только Backend API Kircraft.

---

## 2. Границы ответственности модуля

Payment API отвечает за:
- создание платежа для заказа в допустимом статусе
- формирование параметров платежа на основании данных заказа
- сохранение попытки оплаты во внутренней системе
- возврат клиенту ссылки на страницу оплаты YooKassa
- обработку webhook-уведомлений YooKassa
- синхронизацию статуса платежа через API YooKassa
- перевод заказа в статус `paid` после успешной оплаты
- фиксацию данных об успешной или неуспешной попытке оплаты
- обеспечение идемпотентности операций создания платежа и обработки webhook

Payment API не отвечает за:
- создание заказа
- изменение состава заказа
- расчет стоимости товаров
- расчет и выбор доставки
- административное управление заказами
- прямую работу клиента с API YooKassa

## 3. Связанные сущности

Модуль использует следующие сущности модели данных:
- `payments`

Модуль зависит от следующих сущностей:
- `orders`
- `customers`

Связь с заказом:
- каждый платеж связан с конкретным заказом через `order_id`
- один заказ может иметь несколько попыток оплаты
- в каждый момент времени допускается только одна актуальная нефинальная попытка оплаты
- успешная оплата изменяет статус заказа на `paid`
- неуспешная попытка оплаты не переводит заказ в `paid`

Связь с клиентом:
- платеж создается только в контексте текущего аутентифицированного клиента
- платеж может быть создан только для заказа, принадлежащего текущему клиенту

Связь с внешней системой:
- внешний платеж создается в YooKassa
- внутренний платеж хранит как внутренние данные Kircraft, так и данные внешней системы
- внешний идентификатор платежа хранится в поле `external_payment_id`
- внутренний идентификатор попытки оплаты хранится в поле `payment_attempt_id`

## 4. Правила работы платежа

### 4.1 Общие правила

- Payment API работает только в контексте текущего аутентифицированного клиента, кроме webhook-обработчика YooKassa
- клиент не может создавать платеж для чужого заказа
- клиент не передает сумму платежа напрямую
- сумма платежа всегда формируется backend на основании `orders.total_price`
- все денежные значения в рамках Payment API передаются в валюте `RUB` в минимальных денежных единицах (integer), согласованно с `orders.total_price` и хранением в БД (`schema.dbml`)
- валюта платежа формируется из заказа и не может отличаться от `RUB`
- `payment_id` является уникальным идентификатором внутренней записи платежа
- `payment_attempt_id` является уникальным бизнес-идентификатором попытки оплаты

### 4.2 Правила создания платежа

- платеж создается только для существующего заказа текущего клиента
- платеж может быть создан только для заказа в статусе `awaiting_payment`
- для создания платежа используется итоговая сумма заказа, включая стоимость товаров и доставки
- клиентский запрос на создание платежа должен содержать заголовок `Idempotency-Key`
- при вызове внешнего API YooKassa используется заголовок `Idempotence-Key`
- `Idempotency-Key` и `Idempotence-Key` являются разными сущностями и используются независимо
- одна попытка оплаты (`payment_attempt_id`) должна быть связана с одним `idempotence_key`
- повторная инициация той же попытки оплаты должна использовать тот же `payment_attempt_id` и тот же `Idempotence-Key`
- параметры платежа (сумма, валюта, описание, return_url) формируются backend и не доверяются клиенту
- `return_url` формируется backend и не должен приниматься напрямую от клиента без проверки

### 4.3 Правила повторной инициации оплаты

- если для заказа уже существует актуальная попытка оплаты в нефинальном статусе и `confirmation_url` остается валидным, система возвращает существующую ссылку на оплату
- если предыдущая попытка оплаты имеет финальный статус или ссылка на оплату больше невалидна, система создает новую попытку оплаты
- повторное создание платежа не должно приводить к наличию нескольких активных нефинальных попыток оплаты для одного заказа

### 4.4 Правила статусов платежа

Внутренние статусы попытки оплаты:
- `created` — внутренняя запись попытки оплаты создана, но внешний статус еще не зафиксирован
- `pending` — платеж создан во внешней системе и ожидает завершения пользователем
- `succeeded` — платеж успешно завершен и применен к заказу (заказ переведен в `paid`)
- `canceled` — платеж отменен или не применен к заказу (например, поздняя/устаревшая попытка, результат по которой не должен менять состояние заказа)
- `error` — внутренняя ошибка Kircraft при создании или обработке платежа

Статусы YooKassa, используемые в интеграции:
- `pending`
- `waiting_for_capture`
- `succeeded`
- `canceled`

Статусы `succeeded` и `canceled` являются финальными.  
Статус `pending` является нефинальным.  
Статус `waiting_for_capture` является нефинальным.

**MVP v1 (без отдельного capture-flow):** двухстадийная оплата и операции capture в Payment API **не входят**. Значение `waiting_for_capture` допустимо в `provider_status` как отражение ответа YooKassa; отдельного внутреннего статуса Kircraft под capture **нет** — бизнес-логика опирается на перечень `status` из §4.4 и §7.4 (пока попытка не завершена у провайдера финально, внутренний `status` остается нефинальным, как для `pending`).

Разделение внешнего и внутреннего статуса:
- `provider_status` является источником внешнего состояния платежа в YooKassa
- `status` является внутренним нормализованным статусом Kircraft для бизнес-логики
- переходы статуса заказа определяются по внутреннему `status`, а не напрямую по `provider_status`

### 4.5 Правила синхронизации статусов

- основным источником финального результата оплаты является webhook YooKassa
- если webhook не получен или не может быть обработан, система выполняет fallback-проверку статуса во внешней системе и приводит платеж и заказ в консистентное состояние; эта сверка — **внутренний процесс** (фон, повторные попытки и т.п.), а не часть публичного `GET /api/v1/payments/{payment_id}`
- публичный `GET /api/v1/payments/{payment_id}` возвращает состояние из Kircraft на момент запроса и **не инициирует** обращений к API YooKassa и **не заменяет** fallback-синхронизацию
- обработка webhook и fallback-синхронизация должны быть идемпотентными
- повторная обработка одного и того же webhook не должна приводить к повторному изменению состояния заказа или созданию дублирующих записей
- механизм проверки webhook должен быть реализован до допуска уведомления к обработке
- если статус получен через fallback-проверку, применяются те же правила переходов и валидации, что и при обработке webhook

### 4.6 Правила обработки устаревших попыток оплаты

- при наличии нескольких попыток оплаты система должна учитывать актуальность попытки
- актуальность попытки определяется сравнением `metadata.payment_attempt_id` с текущей активной попыткой оплаты заказа
- успешный результат по устаревшей/late попытке оплаты не должен приводить к `payment.status = succeeded` в бизнес-смысле Kircraft и не должен автоматически менять статус заказа на `paid`
- такие случаи фиксируются как инциденты и требуют дополнительной обработки

### 4.7 Правила обновления заказа
При успешной оплате актуальной попытки:
- статус платежа переводится в `succeeded`
- статус заказа переводится в `paid`
- фиксируется дата и время успешной оплаты

При успешной оплате устаревшей/late попытки (когда результат не должен менять состояние заказа):
- статус платежа не переводится в `succeeded` и заказ не переводится в `paid`
- внутренний статус попытки переводится в `canceled` (платеж не применен к заказу)

При неуспешной оплате:
- статус платежа переводится в `canceled` или `error`
- заказ сохраняет статус `awaiting_payment`, если не применены отдельные правила отмены или таймаута

### 4.8 Правила безопасности

- учетные данные YooKassa хранятся только на стороне Backend API Kircraft
- клиентские интерфейсы не имеют доступа к `shopId` и `secretKey`
- webhook-уведомления должны проходить проверку подлинности и структуры
- неподтвержденный webhook не должен изменять статус платежа или заказа
- данные платежа и заказа должны сопоставляться по `order_id`, сумме, валюте и метаданным

### 4.9 Правила консистентности

- не допускается состояние, при котором платеж имеет статус `succeeded`, а заказ не переведен в `paid`
- не допускается состояние, при котором заказ имеет статус `paid`, но отсутствует успешный платеж
- при возникновении внутренних ошибок система должна фиксировать инцидент и обеспечивать возможность повторной синхронизации

### 4.10 Правила таймаута оплаты

- заказ в статусе `awaiting_payment` может быть автоматически отменен системой по таймауту
- конкретное значение таймаута не фиксируется в данном документе
- механизм таймаута не должен нарушать консистентность между заказом и платежами

## 5. Endpoint overview

Модуль включает следующие методы (префикс API: `/api/v1`):
- `POST /api/v1/payments` — создать новую попытку оплаты или вернуть актуальную существующую попытку (create-or-return)
- `GET /api/v1/payments/{payment_id}` — получить состояние внутренней записи платежа
- `POST /api/v1/payments/webhook/yookassa` — принять webhook-уведомление от YooKassa и применить допустимые переходы

## 6. Контракты методов

### 6.1 Создание или возврат попытки оплаты

#### Метод и путь

`POST /api/v1/payments`

#### Назначение

Инициирует оплату заказа в YooKassa по модели create-or-return:
- возвращает существующую **активную нефинальную попытку**, если она актуальна и `confirmation_url` валиден;
- создает новую попытку, если актуальной попытки нет.

#### Заголовки request

- `Idempotency-Key` — обязательный заголовок клиентского API.

#### Request body

```json
{
  "order_id": "order_001"
}
```

#### Поля request

- `order_id` — обязательное поле, непустая строка (после trim).

#### Правила request

- запрос выполняется только в контексте текущего аутентифицированного клиента;
- заказ должен существовать и принадлежать клиенту;
- допустимый статус заказа для старта оплаты: только `awaiting_payment`;
- сумма платежа вычисляется backend из `orders.total_price` и передается в YooKassa как `amount.value` с двумя знаками после запятой;
- backend обязан различать:
  - `Idempotency-Key` (client API key),
  - `Idempotence-Key` (provider key для запроса в YooKassa);
- одна попытка (`payment_attempt_id`) связана с одним provider `Idempotence-Key`;
- в один момент времени для заказа допускается не более одной активной нефинальной попытки.

#### Правила повторного вызова

Повторный вызов `POST /api/v1/payments` обрабатывается по следующему порядку:
1. Если ключ `Idempotency-Key` уже обрабатывался для того же клиента и того же `order_id`, возвращается ранее согласованный результат.
2. Если для заказа есть актуальная активная нефинальная попытка с валидным `confirmation_url`, возвращается эта попытка (новая не создается).
3. Если активной нефинальной попытки нет (либо она финальная/невалидная), создается новая попытка.
4. Если состояние попыток противоречит инвариантам (например, обнаружено более одной активной нефинальной попытки), возвращается `409 payment_attempt_conflict`.

#### Взаимодействие с YooKassa

При создании новой попытки backend вызывает YooKassa с:
- Basic Auth (`shopId:secretKey`);
- заголовком `Idempotence-Key`;
- `confirmation.type = redirect`;
- `metadata.order_id`;
- `metadata.payment_attempt_id`.

#### Response 201 (создана новая попытка)

```json
{
  "data": {
    "payment_id": "payment_001",
    "payment_attempt_id": "attempt_001",
    "order_id": "order_001",
    "status": "pending",
    "provider_status": "pending",
    "amount": 350000,
    "currency": "RUB",
    "confirmation_url": "https://yookassa.ru/checkout/payments/v2/contract?orderId=example",
    "paid_at": null,
    "expires_at": "2026-03-19T19:30:00Z",
    "created_at": "2026-03-19T18:35:00Z",
    "updated_at": "2026-03-19T18:35:00Z"
  },
  "meta": {
    "request_id": "uuid"
  }
}
```

#### Response 200 (возвращена существующая попытка)

Формат ответа совпадает с `201`.

#### Возможные ошибки

- `400 invalid_request` — некорректная структура/тип данных или отсутствует обязательный `Idempotency-Key`;
- `401 unauthorized`;
- `404 order_not_found`;
- `409 order_not_payable`;
- `409 payment_attempt_conflict`;
- `422 validation_error`;
- `422 item_not_available`;
- `500 payment_create_failed`;
- `502 payment_provider_error`;
- `503 payment_provider_unavailable`;
- `500 internal_error`.

### 6.2 Получение информации о платеже

#### Метод и путь

`GET /api/v1/payments/{payment_id}`

#### Назначение

Возвращает состояние внутренней записи платежа текущего клиента.

#### Правила request

- `payment_id` должен относиться к платежу текущего клиента;
- доступ к чужому платежу не допускается;
- метод **только читает локальное состояние** Kircraft;
- метод **не инициирует** синхронизацию с YooKassa и не заменяет webhook/fallback-процессы.

#### Response 200

```json
{
  "data": {
    "payment_id": "payment_001",
    "payment_attempt_id": "attempt_001",
    "order_id": "order_001",
    "status": "pending",
    "provider_status": "pending",
    "amount": 350000,
    "currency": "RUB",
    "confirmation_url": "https://yookassa.ru/checkout/payments/v2/contract?orderId=example",
    "paid_at": null,
    "expires_at": "2026-03-19T19:30:00Z",
    "created_at": "2026-03-19T18:35:00Z",
    "updated_at": "2026-03-19T18:35:00Z"
  },
  "meta": {
    "request_id": "uuid"
  }
}
```

#### Возможные ошибки

- `401 unauthorized`;
- `404 payment_not_found`;
- `500 internal_error`.

### 6.3 Обработка webhook YooKassa

#### Метод и путь

`POST /api/v1/payments/webhook/yookassa`

#### Назначение

Принимает webhook от YooKassa, валидирует событие, применяет допустимые переходы статусов платежа и заказа.

#### Требования к безопасности

- endpoint доступен без пользовательской аутентификации;
- webhook обязан пройти проверку подлинности;
- при непрохождении проверки возвращается `401 unauthorized_webhook`.

#### Структура payload (обязательные поля)

```json
{
  "event": "payment.succeeded",
  "object": {
    "id": "2b9e...",
    "status": "succeeded",
    "metadata": {
      "order_id": "order_001",
      "payment_attempt_id": "attempt_001"
    }
  }
}
```

Обязательны:
- `object.id`;
- `object.status`;
- `object.metadata.order_id`;
- `object.metadata.payment_attempt_id`.

#### Правила обработки

- обработка webhook идемпотентна;
- дедупликация выполняется по идентификатору события/платежа и финальному статусу;
- допускаются только переходы из нефинальных provider-статусов (`pending`, `waiting_for_capture`) в финальные (`succeeded`, `canceled`);
- переходы из финальных статусов в нефинальные запрещены и приводят к `409 payment_state_conflict`;
- late/устаревшая попытка (неактуальный `payment_attempt_id`) не переводит заказ в `paid`;
- при `succeeded` актуальной попытки:
  - `payment.status -> succeeded`,
  - `order.status -> paid`,
  - фиксируется `paid_at`;
- при `canceled` или неуспешном исходе:
  - `payment.status -> canceled` или `error`,
  - `order.status` остается `awaiting_payment` (если не применены отдельные правила таймаута/отмены).

#### Response 200

```json
{
  "data": {
    "accepted": true
  },
  "meta": {
    "request_id": "uuid"
  }
}
```

#### Возможные ошибки

- `400 invalid_request`;
- `401 unauthorized_webhook`;
- `404 payment_not_found`;
- `409 payment_state_conflict`;
- `500 internal_error`.

## 7. Формат данных и state machine

В контракте ответов не возвращаются технические поля модели, используемые только для внутренней диагностики: `external_payment_id`, `provider_metadata`, `last_webhook_at`, `last_status_source`.

### 7.1 Payment detail (канонический формат)

```json
{
  "payment_id": "payment_001",
  "payment_attempt_id": "attempt_001",
  "order_id": "order_001",
  "status": "pending",
  "provider_status": "pending",
  "amount": 350000,
  "currency": "RUB",
  "confirmation_url": "https://yookassa.ru/checkout/payments/v2/contract?orderId=example",
  "paid_at": null,
  "expires_at": "2026-03-19T19:30:00Z",
  "created_at": "2026-03-19T18:35:00Z",
  "updated_at": "2026-03-19T18:35:00Z"
}
```

### 7.2 Термины

- `Idempotency-Key` — ключ идемпотентности клиентского API `POST /api/v1/payments`.
- `Idempotence-Key` — ключ идемпотентности запроса к YooKassa.
- `payment attempt` — одна внутренняя попытка оплаты (`payment_attempt_id`).
- `active non-final attempt` — попытка в нефинальном состоянии, пригодная для продолжения оплаты.
- `actual attempt` — текущая актуальная попытка заказа, которая может изменить бизнес-состояние заказа.
- `late attempt` — устаревшая попытка, результат которой не должен переводить заказ в `paid`.

### 7.3 State machine

#### Provider statuses (YooKassa)
- `pending`
- `waiting_for_capture`
- `succeeded` (final)
- `canceled` (final)

#### Internal payment statuses (Kircraft)
- `created`
- `pending`
- `succeeded` (final)
- `canceled` (final)
- `error` (final)

#### Order statuses в payment lifecycle
- `awaiting_payment`
- `paid`

#### Допустимые переходы

- Payment (internal):
  - `created -> pending | error`
  - `pending -> succeeded | canceled | error`
  - `succeeded` / `canceled` / `error` — финальные
- Order:
  - `awaiting_payment -> paid` только при `succeeded` **актуальной** попытки
  - результаты late attempt не должны выполнять переход заказа в `paid`

### 7.4 Правила `confirmation_url`

- ключ `confirmation_url` всегда присутствует в формате payment detail;
- для активных нефинальных попыток значение обычно URL (может быть `null`, если ссылка недоступна/просрочена);
- для финальных попыток значение `null`.

## 8. Ошибки модуля

Ошибки соответствуют общему формату `errors.md`.

### 8.1 Классы ошибок

#### 400 invalid_request
- некорректная структура тела;
- отсутствуют обязательные поля;
- неверные типы данных;
- отсутствует обязательный `Idempotency-Key`;
- webhook без обязательных полей `object.*`.

#### 401 unauthorized
- клиент не аутентифицирован.

#### 401 unauthorized_webhook
- webhook не прошел проверку подлинности.

#### 404 order_not_found
- заказ не существует или не принадлежит текущему клиенту.

#### 404 payment_not_found
- платеж не существует или не принадлежит текущему клиенту.

#### 409 order_not_payable
- заказ не допускает инициацию оплаты в текущем статусе.

#### 409 payment_attempt_conflict
- конфликт состояния попыток (например, нарушен инвариант одной активной нефинальной попытки).

#### 409 payment_state_conflict
- входящий transition платежа противоречит state machine.

#### 422 validation_error
- бизнес-валидация не пройдена (например, заказ не готов к оплате по бизнес-правилам).

#### 422 item_not_available
- в заказе есть позиции, недоступные для оплаты на момент инициации.

#### 500 payment_create_failed
- не удалось создать/зафиксировать попытку оплаты.

#### 502 payment_provider_error
- ошибка внешнего платежного провайдера.

#### 503 payment_provider_unavailable
- внешний платежный сервис временно недоступен.

#### 500 internal_error
- непредвиденная внутренняя ошибка.

### 8.2 Ошибки по endpoint

- `POST /api/v1/payments`:
  - `400 invalid_request`
  - `401 unauthorized`
  - `404 order_not_found`
  - `409 order_not_payable`
  - `409 payment_attempt_conflict`
  - `422 validation_error`
  - `422 item_not_available`
  - `500 payment_create_failed`
  - `502 payment_provider_error`
  - `503 payment_provider_unavailable`
  - `500 internal_error`

- `GET /api/v1/payments/{payment_id}`:
  - `401 unauthorized`
  - `404 payment_not_found`
  - `500 internal_error`

- `POST /api/v1/payments/webhook/yookassa`:
  - `400 invalid_request`
  - `401 unauthorized_webhook`
  - `404 payment_not_found`
  - `409 payment_state_conflict`
  - `500 internal_error`

## 9. Правила безопасности и консистентности

Применяются правила из `security.md`.

Дополнительно:
- все клиентские методы требуют аутентификации;
- клиент работает только со своими заказами и платежами;
- сумма/валюта/описание/return_url формируются только backend;
- клиент не управляет status transitions;
- webhook обрабатывается только после проверки подлинности;
- webhook и fallback-синхронизация идемпотентны;
- одновременно не допускается более одной активной нефинальной попытки на заказ;
- поздний успешный результат по late attempt не переводит заказ в `paid`;
- состояние `payment.succeeded` и `order.paid` должно оставаться консистентным.
