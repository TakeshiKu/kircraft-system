# Delivery API

## 1. Назначение модуля

Delivery API предназначен для расчета и выбора параметров доставки на этапе оформления заказа.

Модуль обеспечивает:
- расчет стоимости и сроков доставки
- получение списка доступных пунктов выдачи заказов (ПВЗ)
- выбор конкретного ПВЗ
- фиксацию выбранной доставки до создания заказа
- предоставление текущего выбранного варианта доставки в процессе оформления заказа

Delivery API используется клиентскими интерфейсами системы:
- Web
- Telegram-бот
- Mini App
- другие клиентские каналы

Взаимодействие с внешним провайдером доставки выполняется только серверной частью системы. Клиентские интерфейсы не взаимодействуют с CDEK напрямую.

---

## 2. Границы ответственности модуля

Delivery API отвечает за:
- расчет доставки по введенному пользователем городу
- получение от провайдера доставки списка доступных ПВЗ и параметров доставки
- нормализацию и возврат данных доставки клиентскому интерфейсу
- выбор пользователем конкретного ПВЗ
- сохранение выбранной доставки во временном состоянии оформления заказа
- предоставление текущего выбранного варианта доставки
- проверку наличия валидной доставки перед созданием заказа

Delivery API не отвечает за:
- создание заказа
- создание платежа
- изменение состава корзины
- расчет стоимости товаров
- административное управление доставкой
- прямую работу клиента с API CDEK

---

## 3. Связанные сущности

Модуль зависит от следующих сущностей модели данных:
- `orders` — основное хранилище snapshot выбранной доставки в денормализованных полях `delivery_*` и `pickup_point_*` (источник истины)
- `order_deliveries` — историческая таблица попыток выбора доставки для заказа (multi-attempt log)

Связь с заказом:
- доставка выбирается до создания заказа
- до создания заказа выбранная доставка хранится во временном состоянии оформления заказа
- при создании заказа выбранная доставка фиксируется как snapshot в денормализованных полях `orders.*`
- без выбранной и валидной доставки заказ не может быть создан

Связь с внешней системой:
- в текущей версии поддерживается только провайдер доставки `cdek`
- в текущей версии поддерживается только тип доставки `pickup_point`
- список ПВЗ, стоимость и сроки доставки определяются внешним провайдером

---

## 4. Правила работы доставки

### 4.1 Общие правила

- в текущей версии системы поддерживается только провайдер доставки `cdek`
- выбор провайдера пользователем не поддерживается
- в текущей версии поддерживается только тип доставки `pickup_point`
- доставка выбирается до создания заказа
- до создания заказа доставка существует только во временном состоянии оформления заказа
- после создания заказа доставка фиксируется в денормализованных полях `orders.*`
- без валидной доставки заказ не может быть создан

### 4.2 Правила расчета доставки

- пользователь указывает город доставки
- backend выполняет запрос к провайдеру доставки
- backend получает список доступных ПВЗ и параметры доставки
- backend возвращает клиенту нормализованный список доступных вариантов
- ответ должен содержать:
  - `delivery_provider`
  - `delivery_type`
  - `city`
- каждый вариант доставки должен содержать:
  - `pickup_point_id`
  - `pickup_point_name`
  - `pickup_point_address`
  - `delivery_eta_min_days`
  - `delivery_eta_max_days`
  - `delivery_price` — стоимость доставки в минимальных денежных единицах
  - `delivery_currency`

### 4.3 Правила выбора ПВЗ

- пользователь может выбрать только один ПВЗ из списка, полученного в результате расчета доставки
- после выбора ПВЗ backend сохраняет выбранную доставку во временном состоянии оформления заказа
- выбранная доставка должна включать:
  - город
  - провайдера доставки
  - тип доставки
  - идентификатор ПВЗ
  - название ПВЗ
  - адрес ПВЗ
  - сроки доставки
  - стоимость доставки
  - валюту доставки

### 4.4 Правила пересчета доставки

- изменение города доставки всегда приводит к сбросу выбранной доставки (Selected Delivery = null) и необходимости повторного расчета через `POST /api/v1/delivery/calculate`
- изменение состава корзины (добавление товара, удаление товара или изменение количества) всегда приводит к сбросу выбранной доставки (Selected Delivery = null) и необходимости повторного расчета доставки
- backend имеет право признать ранее рассчитанные варианты доставки неактуальными; в этом случае ранее выбранная доставка аннулируется (Selected Delivery = null) и пользователю требуется выполнить повторный расчет
- при повторном расчете доставки ранее выбранный ПВЗ должен быть сброшен
- после сброса `Selected Delivery = null` и требуется повторный выбор доставки
- до создания заказа система может выполнять несколько расчетов доставки

### 4.5 Правила валидации перед созданием заказа

Перед созданием заказа backend проверяет:
- доставка была рассчитана
- ПВЗ был выбран
- выбранная доставка принадлежит текущему состоянию оформления заказа
- выбранная доставка содержит корректный город
- выбранная доставка содержит корректный идентификатор ПВЗ
- выбранная доставка содержит стоимость доставки
- валюта доставки равна `RUB`
- `delivery_currency` в текущей версии всегда равна `RUB`, задается backend и не передается клиентом
- в будущих версиях `delivery_currency` может отличаться от валюты заказа: в заказе используется нормализованное значение, а пересчет выполняется на стороне backend
- при отличии валюты доставки от валюты заказа backend конвертирует стоимость доставки и записывает уже нормализованное значение в заказ

Если хотя бы одно условие не выполнено:
- заказ не создается
- клиент получает ошибку
- выбранная доставка должна быть рассчитана или выбрана повторно

### 4.6 Правила фиксации доставки в заказе

При создании заказа backend записывает snapshot выбранной доставки в денормализованные поля `orders.*`:
- `delivery_provider`
- `delivery_type`
- `delivery_currency`
- `pickup_point_id`
- `pickup_point_name`
- `pickup_point_address`
- `delivery_eta_min_days`
- `delivery_eta_max_days`
- `delivery_price`
- `city`

Эти поля являются **источником истины** для текущей доставки заказа.

Таблица `order_deliveries` используется как историческая запись попыток выбора доставки для одного заказа (multi-attempt log). Запись в `order_deliveries` создаётся при наличии нескольких последовательных выборов доставки для одного заказа и не является обязательной для основного flow. Статус доставки в `order_deliveries` отдельно как источник истины не используется.

После создания заказа:
- изменение доставки в рамках текущего клиентского API не поддерживается
- источником истины для доставки заказа являются денормализованные поля `orders.*`
- если для заказа существует запись в `order_deliveries`, она трактуется как история попыток выбора и должна быть консистентна с актуальным snapshot в `orders.*`
- `items_total` — сумма всех позиций заказа:
  - с учетом количества
  - с учетом примененных скидок (если есть)
  - значение рассчитывается backend на основе текущего состояния корзины
- итоговая стоимость заказа (`orders.total_price`) рассчитывается как `items_total + delivery_price`
- в модели данных это поле хранится как `orders.total_price`, а в API представляется как `totals.total_price` (одно и то же значение на уровне заказа)
- `delivery_price` участвует в расчете итоговой стоимости заказа на этапе создания заказа и фиксируется на момент создания заказа; изменение после создания заказа не поддерживается

### 4.7 Правила консистентности

- выбранная доставка должна относиться к текущему клиенту и текущему процессу оформления заказа
- стоимость доставки, используемая в итоговой сумме заказа, должна совпадать со стоимостью выбранной доставки
- после создания заказа данные доставки не должны изменяться из-за повторного расчета у внешнего провайдера

### 4.8 Правила расширения

- архитектура модуля допускает добавление других провайдеров доставки в будущем
- архитектура модуля допускает добавление других типов доставки в будущем
- в текущем контракте клиент не выбирает провайдера доставки и не выбирает тип доставки

## 5. API методы

### 5.1 Общая схема

Delivery API предоставляет методы для:
- расчета доставки и получения списка ПВЗ
- выбора конкретного ПВЗ
- получения текущего выбранного варианта доставки

Все методы работают в контексте текущего пользователя и текущего процесса оформления заказа (checkout).

Контекст оформления заказа определяется текущей активной корзиной пользователя.
Изменение состава корзины может требовать повторного расчета доставки, так как выбранная доставка должна соответствовать текущему состоянию checkout.

---

### 5.2 POST /api/v1/delivery/calculate

#### Назначение

Метод используется для расчета доставки по введенному городу и получения списка доступных ПВЗ.

#### Описание

- принимает город доставки
- выполняет расчет доставки через провайдера (CDEK)
- возвращает список доступных ПВЗ с параметрами доставки
- сбрасывает ранее выбранную доставку, если она была установлена

#### Request (пример)

```json
{
  "city": "Москва"
}
```

#### Response (пример)

```json
{
  "data": {
    "delivery_provider": "cdek",
    "delivery_type": "pickup_point",
    "city": "Москва",
    "options": [
      {
        "pickup_point_id": "MSK123",
        "pickup_point_name": "СДЭК на Тверской",
        "pickup_point_address": "г. Москва, ул. Тверская, д. 10",
        "delivery_eta_min_days": 2,
        "delivery_eta_max_days": 4,
        "delivery_price": 35000,
        "delivery_currency": "RUB"
      }
    ]
  }
}
```

#### Особенности

- метод может вызываться многократно до создания заказа
- при каждом вызове возвращаются актуальные данные провайдера
- после каждого вызова `POST /api/v1/delivery/calculate` пересоздается список доступных вариантов доставки, выбранная доставка сбрасывается (Selected Delivery = null) и пользователь обязан заново выбрать вариант доставки
- если список ПВЗ пуст, возвращается ошибка delivery_not_available

---

### 5.3 POST /api/v1/delivery/select

#### Назначение

Метод используется для выбора конкретного ПВЗ и сохранения выбранной доставки.

#### Описание

- принимает идентификатор ПВЗ из последнего расчета доставки
- сохраняет выбранную доставку во временном состоянии оформления заказа

#### Request (пример)

```json
{
  "pickup_point_id": "MSK123"
}
```

#### Response (пример)

```json
{
  "data": {
    "delivery_provider": "cdek",
    "delivery_type": "pickup_point",
    "city": "Москва",
    "pickup_point_id": "MSK123",
    "pickup_point_name": "СДЭК на Тверской",
    "pickup_point_address": "г. Москва, ул. Тверская, д. 10",
    "delivery_eta_min_days": 2,
    "delivery_eta_max_days": 4,
    "delivery_price": 35000,
    "delivery_currency": "RUB"
  }
}
```

#### Особенности

- выбор возможен только из актуального результата последнего вызова `POST /api/v1/delivery/calculate`
- при попытке выбрать ПВЗ вне текущего расчета возвращается ошибка pickup_point_not_found
- повторный выбор того же ПВЗ допустим и не приводит к ошибке

---

### 5.4 GET /api/v1/delivery/current

#### Назначение

Метод используется для получения текущей выбранной доставки в процессе оформления заказа.

#### Описание

- возвращает выбранную доставку, если она была сохранена
- используется клиентом для отображения текущего состояния доставки
- endpoint всегда возвращает HTTP 200
- если доставка не рассчитана или не выбрана, возвращается `data: null` и endpoint **не возвращает ошибки**

#### Response (если доставка выбрана)

```json
{
  "data": {
    "delivery_provider": "cdek",
    "delivery_type": "pickup_point",
    "city": "Москва",
    "pickup_point_id": "MSK123",
    "pickup_point_name": "СДЭК на Тверской",
    "pickup_point_address": "г. Москва, ул. Тверская, д. 10",
    "delivery_eta_min_days": 2,
    "delivery_eta_max_days": 4,
    "delivery_price": 35000,
    "delivery_currency": "RUB"
  }
}
```

#### Response (если доставка не выбрана)

```json
{
  "data": null
}
```

## 6. Общие ошибки

В рамках Delivery API используются стандартные ошибки системы.

Специфичные ошибки модуля:

- invalid_request — некорректный запрос (HTTP 400)
- validation_error — ошибка валидации входных данных (HTTP 422)
- unauthorized — пользователь не аутентифицирован (HTTP 401)
- delivery_not_available — по указанному городу нет доступных вариантов доставки (HTTP 422)
- pickup_point_not_found — ПВЗ не найден или не относится к текущему расчету (HTTP 404)
- delivery_not_calculated — попытка выбора ПВЗ без предварительного расчета (HTTP 409)
- delivery_not_selected — доставка не выбрана (HTTP 422; в клиентском flow создания заказа может возвращаться как `validation_error` в `Order API`)
- internal_error — внутренняя ошибка системы (HTTP 500)

## 7. Формат данных

### 7.1 Delivery Option

Delivery Option — вариант доставки, полученный в результате расчета.

Содержит:

- `pickup_point_id` — идентификатор ПВЗ во внешней системе
- `pickup_point_name` — название ПВЗ
- `pickup_point_address` — адрес ПВЗ
- `delivery_eta_min_days` — минимальный срок доставки
- `delivery_eta_max_days` — максимальный срок доставки
- `delivery_price` — стоимость доставки в минимальных денежных единицах
- `delivery_currency` — валюта доставки

Поля `delivery_provider`, `delivery_type` и `city` в ответе `POST /api/v1/delivery/calculate` возвращаются на верхнем уровне `data`.

---

### 7.2 Selected Delivery

Selected Delivery — выбранный пользователем вариант доставки, сохраненный во временном состоянии оформления заказа.

Содержит:

- `delivery_provider`
- `delivery_type`
- `city`
- `pickup_point_id`
- `pickup_point_name`
- `pickup_point_address`
- `delivery_eta_min_days`
- `delivery_eta_max_days`
- `delivery_price` — стоимость доставки в минимальных денежных единицах
- `delivery_currency`

Особенности:

- формируется на основе `Delivery Option`
- хранится во временном состоянии checkout
- используется при создании заказа
- после создания заказа переносится в `order_deliveries`

---

### 7.3 Delivery State

Delivery State — внутренняя логическая модель состояния доставки в процессе оформления заказа.

Содержит:

- рассчитанные варианты доставки (`Delivery Option[]`)
- выбранный вариант доставки (`Selected Delivery` или `null`)

Особенности:

- существует только до создания заказа
- используется как часть checkout-state
- пересоздается при каждом новом расчете доставки
- при изменении города ранее выбранная доставка сбрасывается

---

## 9. Безопасность и ограничения

### 9.1 Контекст пользователя

- все операции выполняются в контексте текущего аутентифицированного пользователя
- доставка привязана к текущему процессу оформления заказа пользователя

---

### 9.2 Валидация данных

- backend не доверяет данным, переданным клиентом
- идентификатор ПВЗ проверяется относительно последнего расчета доставки
- стоимость доставки не принимается от клиента и определяется только backend
- параметры доставки формируются только на стороне backend

---

### 9.3 Ограничение доступа

- пользователь может работать только со своим текущим состоянием оформления заказа
- доступ к состоянию доставки других пользователей невозможен
- выбор ПВЗ вне текущего расчета запрещен

---

### 9.4 Консистентность

- данные доставки фиксируются на момент создания заказа
- после создания заказа параметры доставки не должны изменяться автоматически
- повторные расчеты доставки не влияют на уже созданные заказы

---

### 9.5 Ограничения текущей версии

- поддерживается только провайдер доставки `cdek`
- поддерживается только тип доставки `pickup_point`
- пользователь не может изменить провайдера доставки
- пользователь не может изменить тип доставки
