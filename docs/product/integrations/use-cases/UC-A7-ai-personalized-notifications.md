## UC-A7 Формирование персонализированного уведомления клиенту через n8n + LLM

| Раздел | Содержание |
|------|-----------|
| Код | UC-A7 |
| Название | Формирование персонализированного уведомления клиенту через n8n + LLM |
| Описание | Система формирует персонализированный текст уведомления клиенту по событию изменения статуса заказа. Используется утверждённый мастером шаблон и параметры тона; LLM подставляет в шаблон контекст конкретного заказа |
| Акторы | Система (автоматический агент), Notification Module, n8n, LLM Provider, Клиент (получатель) |
| Предусловия | Заказ существует. Произошло событие изменения статуса заказа. Шаблон уведомления и параметры тона для данного типа события утверждены мастером заранее. n8n и LLM Provider доступны |
| Основной сценарий | 1. Система фиксирует событие изменения статуса заказа. |
|  | 2. Notification Module формирует событие уведомления и публикует webhook в n8n. |
|  | 3. n8n принимает событие и определяет тип уведомления по статусу заказа. |
|  | 4. n8n выбирает утверждённый мастером шаблон и параметры тона для данного типа события. |
|  | 5. n8n запрашивает у LLM Provider персонализированный текст по шаблону и контексту заказа. |
|  | 6. Вызов LLM логируется: промпт, модель, версия, timestamp, результат, статус. |
|  | 7. n8n получает сформированный текст уведомления. |
|  | 8. n8n доставляет уведомление клиенту через канал заказа (например, Telegram). |
|  | 9. Система фиксирует факт отправки уведомления. |
| Альтернативные сценарии | **4a. Шаблон для типа события не утверждён** |
|  | 4a1. n8n использует статический шаблон по умолчанию для данного типа события. |
|  | 4a2. Вызов LLM не выполняется. |
|  | 4a3. Событие отсутствия утверждённого шаблона логируется. |
|  | 4a4. Сценарий продолжается с шага 8. |
|  |  |
|  | **5a. LLM Provider недоступен или вернул ошибку** |
|  | 5a1. n8n переходит на статический шаблон по умолчанию. |
|  | 5a2. Сбой LLM логируется. |
|  | 5a3. Сценарий продолжается с шага 8. |
| Исключения | n8n недоступен — Notification Module фиксирует ошибку доставки события и работает в режиме повторной отправки в соответствии с политикой ретраев. |
|  | Канал доставки клиента недоступен — n8n фиксирует ошибку доставки уведомления клиенту. |
| Постусловия | Клиент получил уведомление: либо персонализированное (через LLM), либо по статическому шаблону (graceful degradation). Вызов LLM (при наличии) зафиксирован в логе AI-вызовов. Данные заказа AI не изменялись |

Связанные BR: BR-AI-03, BR-AI-04, BR-AI-05, BR-AI-06, BR-14.
