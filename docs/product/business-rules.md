# Business Rules — Kircraft System

## Назначение документа

Документ описывает бизнес-правила системы Kircraft.  
Бизнес-правила определяют ограничения поведения системы и пользователей, которые должны соблюдаться при выполнении пользовательских сценариев.

Правила используются при:

- разработке backend-логики
- описании Use Cases
- разработке интерфейсов
- тестировании системы

---

# Перечень бизнес-правил

| Код | Правило | Условие / Ограничение | Связанные Use Cases |
|----|----|----|----|
| BR-01 | Оформление заказа возможно только при наличии товаров в корзине | Система не позволяет перейти к оформлению заказа, если корзина пуста | UC-07, UC-08 |
| BR-02 | Заказ создается только после подтверждения оформления и полной валидации данных | Если клиент не подтверждает оформление или валидация данных оформления не проходит, заказ не создается | UC-08 |
| BR-03 | Контактные данные для checkout: имя и телефон обязательны, email опционален | Клиент не может завершить оформление заказа без указания имени и номера телефона; отсутствие email не блокирует создание заказа | UC-08 |
| BR-04 | Данные доставки являются обязательными | Клиент должен указать город и адрес доставки перед подтверждением заказа | UC-08 |
| BR-05 | Каждому заказу присваивается уникальный идентификатор | При создании заказа система формирует уникальный идентификатор заказа | UC-08, UC-16 |
| BR-06 | Новый заказ получает статус `awaiting_payment` | После создания заказ фиксируется системой со статусом `awaiting_payment` | UC-08 |
| BR-07 | Заказ может быть рассмотрен представителем мастерской после создания | Представитель мастерской может запросить уточнение (→ `needs_clarification`) или отклонить заказ (→ `rejected`); в остальных случаях статус заказа остается `awaiting_payment` | UC-09, UC-10, UC-A1 |
| BR-08 | Оплата доступна только когда заказ находится в `awaiting_payment` | Клиент может перейти к оплате только после перевода заказа в статус `awaiting_payment` | UC-09, UC-15 |
| BR-09 | Клиент может отменить заказ только до оплаты | Отмена заказа клиентом возможна в статусах `draft`, `awaiting_payment` и `needs_clarification` | UC-17 |
| BR-10 | Изменение статуса заказа возможно только по допустимым переходам | Система ограничивает переходы между статусами заказа в соответствии с моделью состояний | UC-09, UC-10, UC-11 |
| BR-11 | Заказ может перейти в статус `paid` только после подтверждения платежа | Подтверждение оплаты выполняется только после подтверждения от платежного провайдера | UC-15 |
| BR-12 | Заказ может перейти в статус `in_progress` только после оплаты | Представитель мастерской может взять заказ в работу только после получения оплаты | UC-A3 |
| BR-13 | Заказ может быть отправлен только после начала выполнения | Отправка заказа возможна только для заказов в статусе `in_progress` | UC-A4 |
| BR-14 | Клиент получает уведомления при изменении статуса заказа | Система отправляет уведомления клиенту при создании/готовности к оплате (переход в `awaiting_payment`), оплате, начале работы, отправке и запросе уточнения | UC-09, UC-10, UC-11 |
| BR-15 | Клиент может просматривать только свои заказы | В разделе «Мои заказы» отображаются только заказы текущего клиента | UC-16 |
| BR-16 | Изменение корзины пересчитывает стоимость | При изменении количества товаров или состава корзины система автоматически пересчитывает итоговую стоимость | UC-07 |
| BR-17 | Уточнение заказа переводит его на повторное рассмотрение | После получения уточнений заказ возвращается на этап рассмотрения | UC-10, UC-A1 |
| BR-18 | После оплаты состав заказа не может быть изменен | Система не позволяет изменять состав оплаченного заказа | UC-15, UC-16, UC-17 |

# Дополнительные правила оплаты

| Код | Правило | Условие / Ограничение | Связанные Use Cases |
|----|----|----|----|
| BR-19 | Один заказ может иметь несколько попыток оплаты | Каждая попытка оплаты фиксируется как отдельная запись платежа | UC-15 |
| BR-20 | Одновременно может существовать только одна активная попытка оплаты | Система должна предотвращать создание нескольких параллельных платежей для одного заказа | UC-15 |
| BR-21 | Подтверждение оплаты выполняется только системой | Возврат пользователя с платежной страницы (`return_url`) не является подтверждением оплаты | UC-15 |
| BR-22 | Успешная оплата фиксируется системой | После подтверждения платежа система переводит заказ в статус `paid` и фиксирует время оплаты | UC-15 |

# Правила неизменяемости данных заказа

| Код | Правило | Условие / Ограничение | Связанные Use Cases |
|----|----|----|----|
| BR-23 | Данные заказа фиксируются в момент оформления | После создания заказа система сохраняет snapshot данных товаров и параметров | UC-08 |
| BR-24 | Изменение каталога не влияет на оформленные заказы | Изменения товаров, параметров и цен не должны изменять уже созданные заказы | UC-08 |

# Правила работы с AI

| Код | Правило | Условие / Ограничение | Связанные Use Cases |
|----|----|----|----|
| BR-AI-01 | AI-сгенерированные описания товаров не публикуются автоматически, требуют апрува мастера | Описание товара становится доступным клиенту только после явного подтверждения мастера в бэкофисе | UC-A5 |
| BR-AI-02 | AI-summary переписки помечается как сгенерированный AI и не заменяет фактическую переписку | Summary отображается с маркировкой источника; фактическая переписка остаётся доступной в неизменном виде | UC-A6 |
| BR-AI-03 | Все вызовы LLM логируются | Каждый вызов сохраняется с полями: промпт, модель, версия модели, timestamp, результат, статус | UC-A5, UC-A6, UC-A7 |
| BR-AI-04 | При недоступности LLM функции AI отключаются | Мастер продолжает работу в обычном режиме без блокировок; интерфейс отображает невозможность AI-операции; уведомления клиенту отправляются по статическому шаблону | UC-A5, UC-A6, UC-A7 |
| BR-AI-05 | AI-уведомления формируются по утверждённым мастером шаблонам и тону | Шаблоны и параметры тона согласуются мастером заранее; LLM подставляет в утверждённый шаблон контекст конкретного заказа | UC-A7 |
| BR-AI-06 | AI не имеет права изменять данные заказа | AI-операции работают в режиме чтения данных заказа; результат AI — вспомогательный контент, не заменяет фактические данные | UC-A6, UC-A7 |
