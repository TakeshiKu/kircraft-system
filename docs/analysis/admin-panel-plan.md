# План подготовки к проектированию и разработке админпанели

## Назначение

Документ фиксирует перечень артефактов, решений и шагов, необходимых для перехода к проектированию и разработке веб-админпанели системы Kircraft.

Документ используется как контрольный список готовности и последовательности работ.

---

# 1. Цель этапа

Цель этапа — подготовить достаточную аналитическую и архитектурную базу для проектирования и разработки веб-админпанели, которая будет использоваться для управления каталогом, заказами, платежами и внутренними действиями системы.

Результатом этапа должна стать минимально жизнеспособная веб-админпанель, которую можно расширять без перепроектирования ядра системы.

---

# 2. Базовый принцип

Админпанель проектируется как отдельный веб-интерфейс поверх общей системы Kircraft.

Это означает:

- база данных едина для всех интерфейсов
- бизнес-логика системы едина
- Telegram-бот, Mini App, сайт и админпанель используют общее системное ядро
- интерфейсы могут меняться и дополняться без перепроектирования модели данных

---

# 3. Что должно быть готово до начала проектирования админпанели

## 3.1. Архитектурные решения

До начала проектирования должны быть зафиксированы следующие решения:

- система проектируется независимо от интерфейса
- Telegram-бот является текущим клиентским каналом, но не ядром системы
- админпанель является отдельным веб-интерфейсом
- платежный провайдер зафиксирован: ЮKassa
- жизненный цикл заказа и жизненный цикл платежа разделены

## 3.2. Аналитические артефакты

До начала проектирования должны быть готовы:

- Vision
- Use Cases
- Business Rules
- Order State Model
- Data Model
- ER-диаграмма
- Payment Flow
- интеграционный use case оплаты
- data mapping для интеграции с ЮKassa

## 3.3. Модель данных

До начала проектирования должна быть зафиксирована модель данных, включающая:

- каталог товаров
- параметры товаров
- фотографии товаров
- клиентов
- корзины
- заказы
- позиции заказов
- платежи
- историю статусов
- внутренних пользователей

---

# 4. Что нужно определить специально для админпанели

## 4.1. Состав модулей первой версии

Необходимо определить, какие модули входят в первую версию админпанели.

Минимально:

- Заказы
- Товары
- Категории
- Коллекции
- Параметры товаров
- Значения параметров
- Платежи
- Внутренние пользователи

## 4.2. Операции по каждому модулю

Для каждого модуля необходимо зафиксировать:

- просмотр списка
- просмотр карточки
- создание
- редактирование
- смену статуса
- удаление или архивирование
- специальные действия

## 4.3. Внутренние сценарии

Необходимо описать основные внутренние сценарии работы:

- просмотр новых заказов
- подтверждение заказа
- запрос уточнения
- отмена заказа
- перевод заказа в работу
- отправка заказа
- просмотр платежа
- создание товара
- редактирование товара
- загрузка фотографий
- настройка параметров товара
- управление справочниками

## 4.4. Роли и права доступа

Необходимо зафиксировать:

- перечень ролей
- доступные действия по ролям
- ограничения по модулям

Минимально рассмотреть роли:

- master
- manager
- admin

---

# 5. Что является входом в проектирование интерфейса админпанели

Перед началом wireframes / прототипирования должны быть готовы:

- перечень модулей админпанели
- CRUD-матрица по сущностям
- роли и права доступа
- внутренние use cases
- перечень обязательных полей в формах
- перечень действий в карточках и списках
- статусная логика для заказов и платежей

---

# 6. Что является входом в разработку админпанели

Перед началом разработки должны быть готовы:

- финальная ER-модель
- финальный состав сущностей и полей
- схема API или понимание backend-методов
- перечень экранов админпанели
- перечень таблиц, форм и карточек
- правила валидации данных
- правила загрузки и хранения фотографий
- правила смены статусов
- сценарий интеграции с ЮKassa

---

# 7. Рекомендуемая последовательность работ

## Шаг 1. Зафиксировать ядро системы
Должны быть завершены и согласованы:

- модель данных
- бизнес-правила
- статусы заказа
- статусы платежа

## Шаг 2. Зафиксировать платежный контур
Должны быть описаны:

- UC-15 Оплата заказа
- интеграционный use case оплаты
- payment flow
- data mapping с ЮKassa

## Шаг 3. Зафиксировать внутренний контур
Должны быть описаны:

- модули админпанели
- внутренние сценарии
- роли и права
- CRUD-матрица

## Шаг 4. Спроектировать интерфейс админпанели
Нужно подготовить:

- список экранов
- таблицы
- карточки
- формы
- переходы
- состав действий по каждому экрану

## Шаг 5. Подготовить backend-контур
Нужно определить:

- какие сущности доступны через API
- какие действия поддерживаются
- какие операции требуют валидации
- какие действия меняют статусы

## Шаг 6. Разработать минимальную версию админпанели
Первая версия должна покрывать:

- управление заказами
- управление товарами
- управление справочниками
- просмотр платежей

## Шаг 7. Расширять без перепроектирования
После запуска первой версии допускается поэтапное расширение:

- дополнительные роли
- дополнительные фильтры
- массовые операции
- аналитика
- расширенное управление каталогом
- дополнительные клиентские интерфейсы

---

# 8. Минимальный набор артефактов, после которого можно переходить к админпанели

Переход к проектированию и разработке админпанели считается допустимым, если готовы:

- финальная модель данных
- ER-диаграмма
- business rules
- state model заказа
- payment flow
- интеграционный use case оплаты
- перечень модулей админпанели
- CRUD-матрица
- роли и права
- внутренние сценарии

---

# 9. Что можно не делать до первой версии админпанели

До первой версии допускается не детализировать:

- аналитику и отчеты
- сложный складской учет
- массовые операции
- расширенную CRM
- историю изменений всех справочников
- сложную матрицу прав доступа

---

# 10. Итог

Админпанель должна разрабатываться не как временный интерфейс управления для Telegram-магазина, а как полноценный внутренний веб-интерфейс системы Kircraft, который может расширяться и использовать общее системное ядро независимо от клиентского канала взаимодействия.
