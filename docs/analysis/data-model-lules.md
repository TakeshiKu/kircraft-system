# Правила модели данных Kircraft

Документ описывает ограничения, инварианты и правила согласованности
модели данных системы Kircraft.

Эти правила дополняют физическую схему базы данных,
описанную в `database/schema.dbml`.

Ограничения могут реализовываться:

- внешними ключами
- уникальными индексами
- check-ограничениями
- логикой backend-сервиса
- комбинацией перечисленных механизмов

---

## 1. Общие правила

1. Каждая сущность имеет уникальный первичный идентификатор.

2. Все связи между сущностями должны соответствовать
внешним ключам, определенным в `schema.dbml`.

3. Записи не должны ссылаться на несуществующие сущности.

4. Поля `created_at` фиксируют момент создания записи.

5. Поля `updated_at` фиксируют момент последнего изменения записи.

---

## 2. Статусы и перечисления

### Каталог

- `categories.status` допускает только:
  - `draft`
  - `active`
  - `archived`

- `collections.status` допускает только:
  - `draft`
  - `active`
  - `archived`

- `products.status` допускает только:
  - `draft`
  - `active`
  - `archived`

### Корзина

- `carts.status` допускает только:
  - `active`
  - `converted`
  - `abandoned`

### Заказы

- `orders.status` допускает только:
  - `created`
  - `needs_clarification`
  - `awaiting_payment`
  - `paid`
  - `in_progress`
  - `shipped`
  - `cancelled`
  - `rejected`

### Внутренние пользователи

- `staff_users.role` допускает только:
  - `master`
  - `manager`
  - `admin`

### Платежи

- `payments.status` допускает только:
  - `created`
  - `pending`
  - `succeeded`
  - `canceled`
  - `error`

- `payments.provider_status` допускает только:
  - `pending`
  - `waiting_for_capture`
  - `succeeded`
  - `canceled`

- `payments.last_status_source` допускает только:
  - `create_response`
  - `webhook`
  - `poll`
  - `manual`

### Внешние учетные записи клиента

- `customer_external_accounts.provider` допускает только:
  - `telegram`
  - `web`
  - `mini_app`
  - `max`
  - `other`

### Канал создания заказа

- `orders.source_channel` допускает только:
  - `telegram`
  - `web`
  - `mini_app`
  - `max`
  - `other`

---

## 3. Правила каталога

1. Каждый `Product` должен принадлежать существующей `Category`.

2. `Collection` для товара является необязательной.

3. `products.base_price >= 0`.

4. У одного товара может быть несколько фотографий,
но не более одной основной фотографии (`is_main = true`).

5. Каждый параметр, доступный для товара,
должен быть объявлен через `product_available_parameters`.

6. Допустимые значения параметров товара
должны задаваться через `product_available_parameter_values`.

---

## 4. Правила параметров товаров

1. Каждый `product_parameter_values.parameter_value_id`
должен принадлежать существующему `product_parameters.parameter_id`.

2. В рамках одного `parameter_id`
значение `value_name` должно быть уникальным.

3. Пара `(product_id, parameter_id)` в `product_available_parameters`
должна быть уникальной.

4. Пара `(product_available_parameter_id, parameter_value_id)`
в `product_available_parameter_values`
должна быть уникальной.

5. `parameter_value_id`, используемый в `product_available_parameter_values`,
должен принадлежать тому же параметру,
который связан с `product_available_parameter_id`.

---

## 5. Правила клиентов

1. Один `Customer` может иметь несколько `CustomerExternalAccount`.

2. Пара `(provider, external_user_id)` в `customer_external_accounts`
должна быть уникальной.

3. Внешняя учетная запись должна ссылаться
на существующего клиента.

4. `customer_id` является главным внутренним идентификатором клиента
во всей системе.

5. Архитектура системы не должна завязываться
на конкретный внешний идентификатор канала
как на главный идентификатор клиента.

6. Для одного клиента в одном канале
может существовать только одна основная учетная запись (`is_primary = true`).

7. Поле `customers.email` является необязательным:
отсутствие email не должно блокировать оформление заказа.

---

## 6. Правила корзины

1. Каждая корзина должна принадлежать одному клиенту.

2. У одного клиента может быть не более одной корзины
в статусе `active`.

3. Позиция корзины (`cart_items`) должна ссылаться
на существующую корзину и существующий товар.

4. Пара `(cart_id, product_id)` в `cart_items`
должна быть уникальной.

5. `cart_items.quantity >= 1`.

6. Каждое выбранное значение параметра в корзине
должно ссылаться на существующую позицию корзины,
существующий параметр и существующее значение параметра.

7. Пара `(cart_item_id, parameter_id)` в `cart_item_selected_values`
должна быть уникальной.

8. `parameter_value_id` в `cart_item_selected_values`
должен принадлежать `parameter_id`.

9. Выбранный параметр и выбранное значение параметра в корзине
должны быть допустимы для конкретного товара
через `product_available_parameters`
и `product_available_parameter_values`.

---

## 7. Правила заказов

1. Каждый `Order` должен принадлежать одному клиенту.

2. Заказ может быть создан из корзины,
но ссылка на корзину не является обязательной.

3. Если `orders.cart_id` заполнен,
заказ должен ссылаться на существующую корзину.

4. Позиции заказа (`order_items`)
должны принадлежать существующему заказу.

5. `order_items.quantity >= 1`.

6. `order_items.price_snapshot >= 0`.

7. В заказе должны храниться snapshot-данные:

- имя клиента
- телефон клиента
- email клиента (если указан)
- название товара
- цена товара
- названия параметров
- значения параметров

8. Snapshot-данные фиксируются на момент оформления заказа
и не должны пересчитываться из каталога после создания заказа.

9. Поле `orders.email_snapshot` является необязательным:
если email не указан при checkout, поле может быть `null`.

10. Изменение товара, параметра или значения параметра
не должно менять уже оформленные заказы.

11. `orders.items_total >= 0`.

12. `orders.delivery_price >= 0`.

13. `orders.total_price >= 0`.

14. `orders.total_price = orders.items_total + orders.delivery_price`.

15. `orders.paid_at` должно быть заполнено
только для подтвержденно оплаченного заказа.

16. Пара `(order_item_id, parameter_id)` в `order_item_selected_values`
должна быть уникальной.

17. `parameter_value_id` в `order_item_selected_values`
должен принадлежать `parameter_id`.

18. Выбранный параметр и выбранное значение параметра в заказе
должны быть допустимы для товара на момент оформления заказа.

19. Если заказ имеет статус `paid`,
поле `orders.paid_at` должно быть заполнено.

---

## 8. Правила истории статусов заказа

1. Каждая запись `order_status_history`
должна ссылаться на существующий заказ.

2. Если указан `changed_by_staff_user_id`,
он должен ссылаться на существующего сотрудника.

3. `from_status` может быть `null`
только для первой записи истории статусов заказа.

4. `changed_by_staff_user_id` может быть `null`,
если изменение статуса выполнено автоматически системой.

5. Значения `from_status` и `to_status`
должны соответствовать допустимым значениям `orders.status`.

6. Записи истории должны отражать фактическую последовательность
изменения статусов заказа во времени.

---

## 9. Правила внутренних пользователей

1. `staff_users.telegram_user_id` должен быть уникальным,
если поле заполнено.

2. Внутренний пользователь может быть деактивирован,
но исторические ссылки на его действия должны сохраняться.

---

## 10. Правила платежей

1. Каждая запись `Payment`
соответствует одной попытке оплаты.

2. Каждый `Payment` должен ссылаться
на существующий заказ.

3. Каждый `Payment` должен ссылаться
на существующего клиента.

4. `payments.customer_id` должен совпадать
с `orders.customer_id` для того же `order_id`.

5. Один заказ может иметь несколько попыток оплаты.

6. У одного заказа не должно быть более одной активной
нефинальной попытки оплаты одновременно.

7. `payments.amount > 0`.

8. `payments.currency` хранит код валюты
по ISO 4217 длиной 3 символа.

9. `payments.idempotence_key` должен быть уникальным.

10. `payments.external_payment_id` должен быть уникальным,
если поле заполнено.

11. `payments.description` не должна превышать 128 символов.

12. `payments.paid_at` должно быть заполнено
только для успешной попытки оплаты.

13. `payments.provider_status` хранит внешний статус YooKassa.

14. `payments.status` хранит внутренний статус попытки оплаты в Kircraft.

15. `return_url` не является подтверждением успешной оплаты.

16. Подтверждение успешной оплаты допускается
только на основании:
- webhook от платежного провайдера
- дополнительной проверки через API провайдера

17. Если попытка оплаты признана успешной,
связанный заказ должен быть переведен в статус `paid`
и должен получить значение `orders.paid_at`.

18. Если попытка оплаты завершилась неуспешно,
заказ не должен автоматически считаться оплаченным.

19. `provider_response_raw` используется только
для трассировки и диагностики
и не является источником бизнес-логики сам по себе.

20. `provider_metadata` должно использоваться
как вспомогательные данные интеграции,
а не как единственный источник идентификации заказа или клиента.

---

## 11. Индексы и технические ограничения

Ниже перечислены рекомендуемые технические ограничения,
которые должны быть реализованы в физической схеме
или на уровне backend:

### Индексы

- индекс по `order_status_history.order_id`
- индекс по `payments.order_id`
- индекс по `payments.customer_id`
- индекс по `cart_items.cart_id`
- индекс по `order_items.order_id`
- индекс по `orders.customer_id`
- индекс по `carts.customer_id`

### Уникальные ограничения

- `customer_external_accounts(provider, external_user_id)`
- `product_available_parameters(product_id, parameter_id)`
- `product_available_parameter_values(product_available_parameter_id, parameter_value_id)`
- `cart_items(cart_id, product_id)`
- `cart_item_selected_values(cart_item_id, parameter_id)`
- `order_item_selected_values(order_item_id, parameter_id)`
- `payments.idempotence_key`
- `payments.external_payment_id` (если не null)
- `staff_users.telegram_user_id` (если не null)

### Дополнительные бизнес-ограничения

- не более одной основной фотографии товара (`is_main = true`) на один товар
- не более одной активной корзины на клиента
- не более одной активной нефинальной попытки оплаты на заказ
