
# Payment UI Flow (Kircraft + YooKassa)

## Назначение

Документ описывает пользовательский flow оплаты на стороне клиентского интерфейса.
Это UI-поток, согласованный с текущими `Order API` и `Payment API`.

---

## Основной UI flow

1. Пользователь находится на экране заказа в статусе `awaiting_payment` и нажимает кнопку **«Перейти к оплате»**.
2. Клиентский интерфейс вызывает `POST /payments` для текущего заказа.
3. Backend:
   - создает новую попытку оплаты, либо
   - возвращает актуальную существующую нефинальную попытку (если она еще валидна).
4. Backend возвращает `confirmation_url`.
5. Клиентский интерфейс перенаправляет пользователя на страницу YooKassa по `confirmation_url`.
6. На стороне YooKassa пользователь:
   - успешно завершает оплату, или
   - отменяет/закрывает оплату.
7. Backend получает финальный результат через webhook YooKassa (и при необходимости через внутреннюю синхронизацию со внешней системой).
8. Если оплата подтверждена, заказ переводится в `paid`.
9. Клиентский интерфейс показывает актуальный статус заказа на основании данных Kircraft.

---

## Важные ограничения flow

- В UI flow используется кнопка **«Перейти к оплате»**.
- В текущем flow нет промежуточного checkout-экрана **«Ожидает оплаты»**.
- В UI не используется вымышленный endpoint вида `POST /orders/{id}/pay`; для инициации оплаты используется `POST /payments`.
- Переход заказа в `paid` происходит только после подтвержденного успешного результата оплаты.
