# YooKassa Payment Integration

## 1. Общее описание интеграции

Интеграция с YooKassa используется в системе Kircraft для обработки онлайн-платежей клиентов при оформлении заказа через клиентские интерфейсы системы.

Клиентские интерфейсы системы могут включать:

- Telegram-бот
- Web-интерфейс
- Telegram Mini App
- другие интерфейсы взаимодействия с пользователем.

Непосредственное взаимодействие с API YooKassa выполняется только серверной частью системы Kircraft (Backend API). Клиентские интерфейсы не взаимодействуют с YooKassa напрямую.

В рамках интеграции используется REST API YooKassa.

Основной сценарий интеграции включает:

1. Создание платежа через метод POST /payments.
2. Получение объекта платежа и ссылки на страницу оплаты (confirmation_url).
3. Перенаправление пользователя на страницу оплаты YooKassa.
4. Выполнение пользователем платежа.
5. Возврат пользователя на страницу системы Kircraft.
6. Получение системой статуса платежа через уведомления YooKassa или запрос GET /payments/{payment_id}.

После успешного завершения платежа система Kircraft фиксирует результат оплаты и переводит заказ клиента в соответствующий статус.

Интеграция реализуется по серверной схеме:

Клиентский интерфейс → Backend API Kircraft → YooKassa API → страница оплаты YooKassa → Backend API Kircraft → Клиентский интерфейс

Это обеспечивает безопасность обработки платежей, так как учетные данные интеграции и секретные ключи YooKassa хранятся только на серверной стороне системы.

## 2. Авторизация при работе с YooKassa API

API YooKassa использует серверную аутентификацию на основе механизма **HTTP Basic Auth**.

Каждый запрос к API должен содержать учетные данные магазина:

- `shopId` — идентификатор магазина  
- `secretKey` — секретный ключ API

Формат передачи авторизации:

Authorization: Basic base64(shopId:secretKey)

Все обращения к API YooKassa выполняются только со стороны **Backend API Kircraft**.  
Клиентские интерфейсы (Telegram Bot, Web Interface, Telegram Mini App) не взаимодействуют с API YooKassa напрямую и не имеют доступа к учетным данным интеграции.

### Конфигурация Backend API

В конфигурации Backend API должны храниться следующие параметры:

- `YOOKASSA_SHOP_ID`
- `YOOKASSA_SECRET_KEY`
- `YOOKASSA_API_BASE_URL`

Пример базового URL:

https://api.yookassa.ru/v3

Хранение конфигурации определяется архитектурой Backend API и может быть реализовано через переменные окружения, конфигурационный файл или иное защищенное хранилище.

### Дополнительные требования

При выполнении операции создания платежа (`POST /payments`) должен передаваться заголовок:

Idempotence-Key

Заголовок используется для защиты от повторного создания одного и того же платежа при повторной отправке запроса.

## 3. API внешней системы

В рамках интеграции система Kircraft использует REST API платежного сервиса YooKassa.

Официальная документация для разработчиков:  
https://yookassa.ru/developers/api

Взаимодействие с API выполняется через HTTPS-запросы.

Базовый URL API:

https://api.yookassa.ru/v3

В рамках текущего сценария интеграции система Kircraft использует следующие методы API.

### POST /payments

Метод используется для создания нового платежа при оформлении заказа пользователем.

Запрос выполняется серверной частью системы Kircraft (Backend API).

В ответе YooKassa возвращает объект платежа, содержащий, в том числе:

- `id` — идентификатор платежа в системе YooKassa  
- `status` — текущий статус платежа  
- `confirmation` — объект подтверждения платежа  
- `confirmation.confirmation_url` — ссылка на страницу оплаты

Полученная ссылка `confirmation_url` используется клиентским интерфейсом Kircraft для перенаправления пользователя на страницу оплаты YooKassa.

### GET /payments/{payment_id}

Метод используется системой Kircraft для получения актуальной информации о платеже.

Запрос выполняется по идентификатору платежа, полученному при создании платежа.

Ответ содержит текущий статус платежа и другие параметры платежной операции.

Полученные данные используются системой Kircraft для синхронизации состояния платежа и обновления статуса заказа.

## 4. Ограничения внешней системы

При интеграции с API YooKassa необходимо учитывать следующие требования и ограничения сервиса.

- Взаимодействие с API осуществляется исключительно по протоколу HTTPS.

- Все запросы должны содержать заголовок `Authorization`, использующий механизм HTTP Basic Auth с параметрами `shopId` и `secretKey`.

- При создании платежа (`POST /payments`) должен передаваться заголовок `Idempotence-Key`, обеспечивающий идемпотентность операции и предотвращающий создание дублирующих платежей.

- Для получения уведомлений о статусе платежей система Kircraft должна предоставить публично доступный URL webhook.

- При работе с API возможны сетевые ошибки, таймауты или временная недоступность сервиса. Система Kircraft должна корректно обрабатывать такие ситуации и предусматривать повторные попытки синхронизации статуса платежа.

- Внешняя система может применять ограничения на частоту запросов (rate limiting), которые необходимо учитывать при проектировании интеграции.
___

### Статусы платежа YooKassa

В рамках текущей интеграции используется схема автоматического списания средств (`capture = true`).

В этом режиме используются следующие статусы платежа:

- `pending` — платеж создан, пользователь еще не завершил оплату;
- `succeeded` — платеж успешно завершен;
- `canceled` — платеж отменен или не был завершен.

Статус `pending` является нефинальным.  
Статусы `succeeded` и `canceled` являются финальными.

## 5. Сценарий работы (Use Case)

### Системы

В процессе выполнения сценария участвуют следующие системы:

- Клиент (пользователь системы Kircraft)
- Клиентский интерфейс Kircraft (Telegram-бот / Web-интерфейс / Telegram Mini App)
- Backend API Kircraft
- База данных Kircraft
- YooKassa API (Внешняя система)

---

### Входные данные

Сценарий инициируется при запуске процесса оплаты заказа пользователем.

Запрос на оплату передается в Backend API Kircraft и содержит:

- `order_id` — идентификатор заказа в системе Kircraft.

Идентификатор пользователя (`user_id`) определяется системой на основании
механизма аутентификации клиента.

Остальные параметры платежа формируются системой на основании данных заказа:

- сумма платежа;
- валюта платежа;
- описание платежа;
- `return_url` для возврата пользователя после оплаты.

---

### Предусловия

Перед выполнением сценария должны быть выполнены следующие условия:

- пользователь идентифицирован системой Kircraft;
- заказ существует в системе;
- запрос на оплату связан с конкретным заказом и пользователем;
- заказ находится в статусе, допускающем оплату;
- интеграция с YooKassa настроена в системе Kircraft;
- в конфигурации Backend API заданы параметры `shopId` и `secretKey`;
- настроен URL webhook для получения уведомлений от YooKassa;
- Backend API имеет доступ к API YooKassa.

---

### Основной сценарий (Happy Path)

#### 1. Инициация оплаты заказа

Пользователь инициирует оплату заказа через клиентский интерфейс системы Kircraft.

Клиентский интерфейс передает в Backend API Kircraft запрос на оплату заказа с указанием `order_id`.

---

#### 2. Проверка заказа и формирование параметров платежа

Backend API Kircraft:

- определяет пользователя, от имени которого выполняется запрос;
- проверяет существование заказа;
- проверяет, что заказ связан с данным пользователем;
- проверяет, что заказ находится в статусе, допускающем оплату;
- определяет сумму и валюту платежа на основании данных заказа в системе Kircraft;
- формирует описание платежа;
- формирует `return_url`;
- формирует внутренний идентификатор попытки оплаты;
- формирует `metadata` для передачи в YooKassa.

---

#### 3. Создание платежа в YooKassa

Backend API Kircraft формирует запрос:

`POST /payments`

В запрос передаются следующие основные данные:

- `amount.value` — сумма платежа
- `amount.currency` — валюта платежа
- `description` — описание платежа
- `confirmation.type = redirect`
- `confirmation.return_url` — URL возврата пользователя после оплаты
- `capture = true`
- `metadata.order_id` — идентификатор заказа в системе Kircraft
- `metadata.user_id` — идентификатор пользователя в системе Kircraft
- `metadata.payment_attempt_id` — внутренний идентификатор попытки оплаты

Дополнительно в HTTP-заголовке передается:

- `Idempotence-Key`

Запрос отправляется в API YooKassa.

---

#### 4. Получение ответа от YooKassa

YooKassa возвращает объект платежа, содержащий, в том числе, следующие данные:

- `id` — идентификатор платежа в YooKassa
- `status` — текущий статус платежа
- `confirmation.confirmation_url` — ссылка на страницу оплаты
- `created_at` — время создания платежа

При успешном создании платежа YooKassa возвращает объект платежа в актуальном статусе. Для созданного платежа в redirect-сценарии начальный статус обычно `pending`.

---

#### 5. Сохранение данных платежа в системе Kircraft

Backend API Kircraft сохраняет в базе данных сведения о созданном платеже, включая:

- идентификатор заказа
- идентификатор пользователя
- внутренний `payment_attempt_id`
- внешний `payment_id` YooKassa
- статус платежа
- сумму платежа
- валюту
- `confirmation_url`
- время создания платежа
- `idempotence_key`

После сохранения данных в системе фиксируется активная попытка оплаты, связанная с заказом.

При необходимости система может перевести заказ во внутренний статус, отражающий наличие незавершенной попытки оплаты.
---

#### 6. Передача ссылки на оплату клиентскому интерфейсу

Backend API Kircraft возвращает клиентскому интерфейсу ссылку `confirmation_url`, полученную от YooKassa.

---

#### 7. Переход пользователя на страницу оплаты YooKassa

Клиентский интерфейс перенаправляет пользователя на страницу оплаты YooKassa по адресу `confirmation_url`.

Пользователь вводит платежные данные и подтверждает оплату.

Проверка корректности платежных данных и выполнение платежной операции выполняются на стороне YooKassa.

---

#### 8. Получение webhook-уведомления от YooKassa

После завершения платежной операции YooKassa отправляет webhook-уведомление в Backend API Kircraft.

Webhook-уведомление содержит:

- `type = notification`
- `event` — тип события
- `object` — объект платежа в актуальном состоянии

Backend API Kircraft принимает уведомление и после успешной обработки возвращает HTTP 200. При получении объекта в нефинальном статусе система может дождаться уведомления или выполнить повторную проверку статуса через API YooKassa.

---

#### 9. Обработка результата платежа

Backend API Kircraft:

- извлекает данные платежа из `object` webhook-уведомления;
- находит внутреннюю запись платежа по `object.id`;
- сопоставляет платеж с заказом по сохраненной связке и данным `metadata`;
- анализирует тип события и статус платежа;
- при необходимости выполняет дополнительную проверку платежа через метод `GET /payments/{payment_id}` как fallback-механизм синхронизации статуса.

---

#### 10. Обновление статуса платежа и заказа

В рамках основного сценария рассматривается успешная оплата, при которой система получает событие успешного завершения платежа и объект платежа в статусе `succeeded`.

В рамках текущей интеграции используется режим автоматического списания средств (`capture = true`). В этом режиме применяются следующие статусы платежа YooKassa:

- `pending`
- `succeeded`
- `canceled`

Финальным успешным статусом платежа является `succeeded`.

Backend API Kircraft обновляет внутренний статус платежа и статус связанного заказа.

При успешной оплате:

- платеж получает статус `succeeded`;
- заказ переводится в статус `paid`.

### Альтернативные сценарии

#### 7A. Пользователь не завершил оплату

7A1. На шаге 7 пользователь переходит на страницу оплаты YooKassa.  
7A2. Пользователь закрывает страницу оплаты или отменяет операцию.  
7A3. YooKassa не формирует событие успешной оплаты.

Результат:

- платеж остается в нефинальном статусе `pending`;
- заказ остается в состоянии ожидания оплаты;
- финальный статус платежа определяется после получения webhook-уведомления
  или при последующей проверке статуса через API YooKassa.

---

#### 8A. Webhook от YooKassa не получен

8A1. После завершения платежной операции Backend API Kircraft не получает webhook-уведомление.  
8A2. Система фиксирует отсутствие webhook в течение ожидаемого интервала.  
8A3. Backend API Kircraft выполняет проверку статуса платежа через `GET /payments/{payment_id}`.  
8A4. Система получает актуальный статус платежа из API YooKassa.  
8A5. Система продолжает обработку сценария, начиная с шага 9 основного сценария.

Результат:

- статус платежа синхронизируется через API YooKassa.

---

#### 9A. Платеж завершился со статусом `canceled`

9A1. Backend API Kircraft получает объект платежа со статусом `canceled`.  
9A2. Система обновляет внутренний статус платежа на `canceled`.  
9A3. Система не переводит заказ в статус `paid`.  
9A4. Заказ переводится во внутренний статус неуспешной оплаты.

Результат:

- платеж считается неуспешным;
- заказ остается неоплаченным.

### Исключения

#### 2A. Заказ не найден

2A1. На шаге 2 Backend API Kircraft выполняет поиск заказа по `order_id`.  
2A2. Система не находит заказ в базе данных.  
2A3. Система прекращает выполнение сценария.  
2A4. Клиентскому интерфейсу возвращается ошибка «Заказ не найден».

Результат:

- платеж не создается;
- статус заказа не изменяется.

---

#### 2B. Заказ не принадлежит пользователю

2B1. На шаге 2 Backend API Kircraft проверяет связь пользователя и заказа.  
2B2. Система определяет, что заказ принадлежит другому пользователю.  
2B3. Система прекращает выполнение сценария.  
2B4. Клиентскому интерфейсу возвращается ошибка доступа.

Результат:

- платеж не создается;
- статус заказа не изменяется.

---

#### 2C. Заказ находится в статусе, не допускающем оплату

2C1. На шаге 2 Backend API Kircraft проверяет текущий статус заказа.  
2C2. Система определяет, что заказ уже оплачен, отменен или находится в ином статусе, при котором создание нового платежа недопустимо.  
2C3. Система прекращает выполнение сценария.  
2C4. Клиентскому интерфейсу возвращается сообщение о невозможности оплаты заказа в текущем статусе.

Результат:

- новый платеж не создается;
- статус заказа не изменяется.

---

#### 3A. Ошибка создания платежа в YooKassa

3A1. На шаге 3 Backend API Kircraft отправляет запрос `POST /payments`.  
3A2. YooKassa возвращает ошибку или запрос завершается неуспешно.  
3A3. Backend API Kircraft фиксирует ошибку интеграции.  
3A4. Система не подтверждает успешное создание внутренней попытки оплаты до получения корректного результата.  
3A5. Клиентскому интерфейсу возвращается ошибка создания платежа.

Результат:

- создание платежа не подтверждено;
- статус заказа не изменяется.

---

#### 3B. Для заказа уже существует активная попытка оплаты

3B1. На шаге 2 или шаге 3 Backend API Kircraft определяет, что для заказа уже существует активная попытка оплаты в нефинальном статусе.  
3B2. Система не должна создавать новый платеж без явного правила повторной оплаты.  
3B3. Система либо возвращает уже существующий `confirmation_url`, либо отклоняет создание новой попытки оплаты.

Результат:

- предотвращается создание нескольких активных платежей для одного заказа;
- консистентность платежного процесса сохраняется.

---

#### 4A. Ответ YooKassa не содержит `confirmation_url`

4A1. На шаге 4 Backend API Kircraft получает ответ от YooKassa.  
4A2. В объекте платежа отсутствует `confirmation.confirmation_url`.  
4A3. Backend API Kircraft фиксирует ошибку интеграции как некорректный или неполный ответ внешней системы.  
4A4. Система не передает пользователю ссылку на оплату и не продолжает сценарий как успешный.

Результат:

- пользователь не может быть перенаправлен на страницу оплаты;
- требуется повторная попытка создания платежа или техническая диагностика.

---

#### 5A. Ошибка сохранения данных платежа в Kircraft

5A1. На шаге 5 Backend API Kircraft получает корректный ответ от YooKassa.  
5A2. Система пытается сохранить данные платежа в базе данных.  
5A3. Происходит ошибка записи данных.  
5A4. Backend API Kircraft фиксирует внутреннюю ошибку.  
5A5. Система должна предусматривать механизм последующей синхронизации или ручного разбора инцидента.

Результат:

- платеж может существовать во внешней системе;
- внутренняя запись отсутствует или неполна;
- состояние системы требует восстановления консистентности.

---

#### 8B. Webhook не прошел проверку

8B1. На шаге 8 Backend API Kircraft получает webhook-запрос.  
8B2. Система выполняет проверку источника запроса и структуры уведомления.  
8B3. Проверка завершается неуспешно.  
8B4. Система отклоняет обработку webhook и фиксирует инцидент.

Результат:

- статус платежа и заказа не изменяется на основании неподтвержденного уведомления;
- требуется повторная доставка webhook или дополнительная проверка статуса через API YooKassa.

---

#### 8C. Webhook содержит недостаточные данные для обработки

8C1. На шаге 8 Backend API Kircraft получает webhook-уведомление.  
8C2. В уведомлении отсутствуют обязательные данные для сопоставления платежа, например `object.id`, `object.status` или `metadata`.  
8C3. Система фиксирует ошибку обработки webhook.  
8C4. Автоматическое обновление статуса платежа и заказа не выполняется.

Результат:

- требуется дополнительная проверка платежа через API YooKassa или ручная диагностика.

---

#### 8D. Backend API Kircraft не смог корректно подтвердить обработку webhook

8D1. На шаге 8 Backend API Kircraft получает webhook-уведомление.  
8D2. В ходе обработки возникает внутренняя ошибка или система не может завершить обработку уведомления.  
8D3. Backend API Kircraft не возвращает корректный HTTP 200.  
8D4. Система фиксирует инцидент и ожидает повторную доставку webhook от YooKassa.

Результат:

- уведомление будет доставлено повторно;
- система должна обеспечивать идемпотентную обработку повторного webhook.

---

#### 9B. Платеж не найден во внутренней системе

9B1. На шаге 9 Backend API Kircraft извлекает из webhook или ответа API внешний `payment_id`.  
9B2. Система не находит внутреннюю запись платежа по данному идентификатору.  
9B3. Система фиксирует критичный инцидент интеграции.  
9B4. Автоматическое обновление статуса заказа не выполняется.

Результат:

- предотвращается некорректное изменение состояния заказа;
- требуется синхронизация или ручной разбор.

---

#### 9C. Данные платежа не совпадают с внутренними данными заказа

9C1. На шаге 9 Backend API Kircraft сопоставляет данные платежа с внутренней моделью.  
9C2. Система обнаруживает несоответствие, например:
- `metadata.order_id` не совпадает с ожидаемым заказом;
- сумма платежа не соответствует сумме заказа;
- валюта платежа не соответствует ожидаемой валюте;
- `user_id` не совпадает с данными внутренней модели.  
9C3. Система фиксирует инцидент.  
9C4. Автоматическое обновление статуса заказа не выполняется.

Результат:

- предотвращается ошибочное зачисление оплаты не тому заказу;
- требуется дополнительная проверка.

---

#### 9D. Не удалось получить актуальный статус платежа через API YooKassa

9D1. На шаге 9 Backend API Kircraft выполняет дополнительную проверку статуса через `GET /payments/{payment_id}`.  
9D2. Запрос завершается ошибкой, таймаутом или не дает корректного ответа.  
9D3. Система фиксирует ошибку синхронизации.  
9D4. Финальный статус заказа не обновляется до получения достоверного результата.

Результат:

- платеж не считается подтвержденным автоматически;
- требуется повторная попытка синхронизации или ручной разбор.

---

#### 10A. Ошибка обновления статуса заказа после успешной оплаты

10A1. На шаге 10 Backend API Kircraft получает подтверждение успешной оплаты.  
10A2. Система пытается обновить внутренний статус заказа.  
10A3. Происходит ошибка обновления данных.  
10A4. Backend API Kircraft фиксирует критический инцидент.  
10A5. Система должна предусматривать механизм повторной обработки или компенсирующей синхронизации.

Результат:

- платеж успешен во внешней системе;
- внутреннее состояние заказа требует восстановления консистентности.

---

#### 10B. Ошибка обновления статуса платежа после получения финального результата

10B1. На шаге 10 Backend API Kircraft определяет финальный статус платежа.  
10B2. Система пытается сохранить обновленный внутренний статус платежа.  
10B3. Происходит ошибка записи в базе данных.  
10B4. Система фиксирует критический инцидент и не должна переводить заказ в противоречивое состояние.

Результат:

- внешний результат платежа известен;
- внутренняя модель платежа требует восстановления консистентности.

### Постусловия

После завершения сценария система Kircraft находится в одном из согласованных состояний.

1. В системе существует запись о попытке оплаты, связанная с заказом.

2. Запись платежа содержит:
- `payment_id` внешней системы YooKassa;
- `order_id` внутренней системы Kircraft;
- `user_id` пользователя;
- сумму и валюту платежа;
- текущий статус платежа.

3. Состояние заказа согласовано со статусом платежа:

- при `payment.status = succeeded`  
  заказ имеет статус `paid`;

- при `payment.status = canceled`  
  заказ остается неоплаченным;

- при нефинальном статусе платежа  
  заказ остается в статусе ожидания оплаты.

4. Между заказом и платежами существует связь:

- каждый платеж связан с конкретным заказом (`order_id`);
- один заказ может иметь несколько попыток оплаты;
- каждая попытка оплаты имеет собственный идентификатор `payment_attempt_id`;
- внешний платеж YooKassa (`payment_id`) связан с конкретной попыткой оплаты.

5. Обработка уведомлений YooKassa является идемпотентной.

Повторное получение webhook по одному и тому же платежу не приводит к:

- повторному изменению статуса заказа;
- созданию дублирующих записей платежей.

6. В системе отсутствуют противоречивые состояния:

- платеж имеет статус `succeeded`, а заказ не отмечен как оплаченный;
- заказ имеет статус `paid`, но соответствующий платеж отсутствует или имеет иной статус.
