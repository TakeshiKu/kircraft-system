# AI Integration Brief — Kircraft System

Документ-контекст для продолжения работы по добавлению AI-функциональности в проект Kircraft. Содержит зафиксированные решения, обоснования, план работ и стиль артефактов. Создан в результате аналитической сессии с владельцем проекта.

---

## 1. Контекст и цель

**Бизнес-контекст.** Kircraft — реальная небольшая мастерская кожаных изделий (брат владельца проекта). Поток — десятки заказов в месяц. Система проектируется как реальный рабочий инструмент, не демо.

**Внешний контекст работы.** Аналитический пакет проекта параллельно используется владельцем как доказательство компетенций при поиске работы (роль системного аналитика с акцентом на AI-автоматизацию). Поэтому в проекте важно:

- решать реальные задачи мастера (не overengineering)
- демонстрировать зрелое продуктовое мышление про AI (не "напихать LLM везде")
- показать архитектурную проработанность интеграции AI в существующую систему

**Цель этой работы.** Добавить AI как часть продукта на уровне аналитических артефактов и архитектуры. Реализация кода AI-функций — отдельная итерация, не входит в текущую задачу.

## 2. Состояние проекта на момент анализа

**Что было до начала работ:**

- Vision, Business Rules (24+ BR), Order Status Model, Data Model (~900 строк), Use Cases Telegram-бота (715 строк), API Specification (~2700 строк), C4-архитектура в Structurizr DSL, Container/Component диаграммы
- Реализован backend на Node.js/TypeScript с DDD-структурой (modules: cart, delivery, order, payment, product, user; integrations: yookassa, cdek), 681 строка PostgreSQL-схемы
- Связанная репа `kircraft-bot-prototype` — рабочий UI-прототип Telegram-бота на Next.js + UX-документация (3 аудита: терминология, post-order flow, UX корзины)

**Чего не было:**

- AI-функциональность отсутствовала во всех артефактах (Vision, Use Cases, BR, архитектура, код)
- Упоминания AI в README не было

**Вывод.** Проект — сильный аналитико-архитектурный артефакт уровня senior, без AI-составляющей. Задача — добавить AI системно, не разрушая существующую структуру.

## 3. Принятые решения с обоснованием

### 3.1. Состав AI-сценариев

**Берём (активные, реализация в плане):**

| # | Сценарий | Где работает | Режим |
|---|---|---|---|
| 1 | Генерация черновика описания товара | Бэкофис (Admin Panel) | Human-in-the-loop: AI готовит черновик, мастер апрувит |
| 2 | AI-summary переписки в карточке заказа | Бэкофис (Admin Panel) | Read-only: AI читает, не меняет данные |
| 4 | Персонализированные уведомления клиенту через n8n + LLM | Интеграционный слой | Авто по утверждённым шаблонам/тону |

**Не берём (Roadmap):**

- Парсинг свободного запроса клиента в Telegram-боте — для текущего потока 10-20 заказов/мес мастер быстрее ответит сам, плюс риски плохого UX от ошибок NLU
- AI-консультант по каталогу с RAG — большая работа (RAG-сборка, эмбеддинги, актуализация индекса), отдельная итерация
- AI-помощник мастеру по уточняющим вопросам к заказу — вторичный сценарий, добавляется при росте потока

**Обоснование выбора 1+2+4:**

- Покрывают **три зоны**: бэкофис, интеграционный слой, и (через #4) клиентскую сторону опосредованно. Системность видна.
- Все три имеют **низкую сложность** реализации
- №1 и №2 — **бэкофисные**, мастер видит результат AI до того, как клиент его видит → нет рисков плохого UX для клиента
- №4 даёт связку **n8n + LLM** — это конкретно ценно для демонстрации "no-code/low-code AI-автоматизация"
- Демонстрируют **разные паттерны интеграции AI**: создание контента (HITL), чтение/анализ (read-only), интеграционная оркестрация

### 3.2. Принципы работы с AI (зафиксированы в Vision)

1. **Human-in-the-loop для контента, идущего клиенту.** Описания товаров публикуются только после подтверждения мастера. Шаблоны и тон AI-уведомлений утверждаются мастером заранее.
2. **AI не меняет данные заказов.** Summary переписки — read-only вспомогательный инструмент.
3. **Маркировка AI-результатов.** Сгенерированные тексты и summary помечаются как AI-сгенерированные.
4. **Логирование вызовов LLM.** Каждый вызов сохраняется (промпт, модель, версия, timestamp, результат).
5. **Graceful degradation.** При недоступности LLM функции AI отключаются, мастер работает в обычном режиме без блокировок.
6. **Без vendor lock-in.** Архитектура не привязана к конкретному LLM-провайдеру.

### 3.3. Архитектурные имена

- **Компонент в Backend API:** `AI Assistant Module` (id в DSL — `aiAssistantModule`)
- **Внешняя система:** `LLM Provider` (id в DSL — `llmProvider`)

**Обоснование:** обе абстракции переживут смену вендора и переезд на локальные модели. Конкретный вендор — деталь реализации, она в `.env.example`, не в архитектурной диаграмме.

### 3.4. LLM-провайдер

**Решено:** облачные LLM (Claude API / GPT API). Локальные не используются — для мастерской на 50-100 заказов/мес локальный инференс — оверкилл по железу и админству. В архитектуре зафиксировано как абстракция `LLM Provider` без привязки к вендору.

### 3.5. Связь с существующими интеграциями

**n8n уже был в архитектуре** как платформа интеграций. Сценарий №4 (AI-уведомления) использует существующую связку `Notification Module → n8n` и добавляет связь `n8n → LLM Provider` для обогащения контента уведомлений.

**Почему уведомления через n8n, а не через AI Assistant Module:** уведомления — это интеграционный поток (триггер события заказа → внешняя оркестрация → доставка сообщения), он логически принадлежит интеграционному слою. AI Assistant Module отвечает за внутренние AI-операции бэкофиса.

**Use Cases (`product/tg-bot/use-cases.md` как образец):**

- Таблица: Код | Правило | Условие/Ограничение | Связанные Use Cases

- Группировка по тематическим секциям

**Архитектура (DSL):**
- Имена компонентов на английском в Title Case
- Описания на русском
- id переменных в camelCase
- Связи описывать действиями ("Запрашивает", "Логирует", "Публикует события")

**Документы вообще:**
- Markdown с разделителями `---`
- Короткие маркированные списки вместо длинных абзацев
- Минимум прозы, максимум структуры

## 7. Что НЕ делать

- **Не выдумывать AI-функциональность сверх согласованного списка.** Сценарии 1, 2, 4 — активные. Остальное — Roadmap, без расширения.
- **Не привязывать архитектуру к конкретному LLM-вендору** (Anthropic/OpenAI). Только абстракция `LLM Provider`. Конкретика — в `.env.example` и в коде интеграции.
- **Не ломать существующие use cases, BR, API.** Только дополнения.
- **Не описывать AI как автономного агента.** Везде явно: human-in-the-loop для генерации контента, read-only для анализа, утверждённые мастером шаблоны для автоматизации.
- **Не реализовывать код в этой итерации.** Уровень 4 — отдельная задача после согласования аналитики и архитектуры.
